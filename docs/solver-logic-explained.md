# 求解逻辑详解

这份文档的目标不是“简单介绍一下”，而是把当前仓库里的求解器讲到下面这个程度：

- 你就算完全没有搜索算法基础，也能知道它到底在做什么
- 你能分清“输入、状态、动作、约束、搜索、校验”分别是什么
- 你看到一条钩计划时，能大概猜出求解器为什么会这样算
- 你后面去看源码时，不会觉得每个文件都像黑盒

如果只想先抓住一句话，可以先记住这个总判断：

> 这套求解器不是一下子把终局“猜出来”，而是不断地从当前现场里找出所有合法的单钩动作，再从这些动作里搜索出一条完整、可回放、可校验的计划。

---

## 1. 先把它想成什么问题

如果用最生活化的话来讲，这个问题有点像：

- 你面前有很多排车
- 每辆车都有自己的最终去向
- 但车不是孤立摆在地上的，而是按顺序停在不同股道里
- 你不能随便拿中间那辆，前面挡着的车必须先处理
- 你不能把车塞进放不下的地方
- 你不能走一条被别的车堵住的路线
- 有些车必须先去称重
- 有些车虽然大方向到了大库，但还要分到合法台位
- 最后导出的计划还必须能被严格复盘

所以它不是一个“静态摆放问题”，而是一个**动态搬运问题**。

换句话说，求解器不是在回答：

> 最终每辆车应该在哪？

而是在回答一个更难的问题：

> 从现在这个现场出发，每一步该怎么搬，最后才能合法地把所有车送到要求的位置？

---

## 2. 为什么这个问题不好算

如果只是“把 A 送到 1 号股道，把 B 送到 2 号股道”，那很简单。

真正难的是下面这些东西叠在一起：

### 2.1 车是有顺序的

一条股道上不是只有“有哪些车”，还要关心：

- 哪辆车在最北端
- 哪辆车在中间
- 哪辆车被挡住了

只要顺序变了，后面能做的动作就会完全不一样。

### 2.2 不是想搬谁就搬谁

当前实现里，机车只能从一条股道的**北端**去挂车。

所以能直接拿走的，只能是：

- 最前面 1 辆
- 最前面 2 辆
- 最前面 3 辆
- …

也就是“北端前缀”。

中间那几辆，如果前面还挡着车，就暂时不可能直接拿出来。

### 2.3 目标不只是“到股道”

不同车辆的目标类型不一样：

- 有的目标是某条明确股道
- 有的目标是某个作业区
- 有的目标是大库里的某个精确台位
- 有的虽然是“大库随机停放”，但也不是随便塞，仍然要分到合法台位

### 2.4 路径本身也有约束

不是源股道和目标股道名字一连就算到了，中间还要满足：

- 拓扑上真的连得通
- 中间股道要净空
- 某些分支长度要够
- 经过 `L1` 时列长不能超过 `190m`
- 倒车所需余量要足够

### 2.5 一步错，后面全变

你把一组车先送到临停线，还是先送去机库，后面所有可行动作都会变。

这就是典型的搜索问题：

- 每一步都会制造新的局面
- 每个新局面又会产生新的可行动作
- 分支会迅速变多

所以这套系统必须一边严格做业务约束，一边控制搜索空间。

---

## 3. 先建立一个最重要的心智模型

理解这套求解器，最重要的是下面这个模型：

### 3.1 它不是直接算“终局”

它真正做的是：

1. 看当前状态
2. 生成所有合法的下一钩
3. 试着走其中一钩
4. 得到一个新状态
5. 再继续重复这个过程
6. 直到所有车都满足目标

所以它更像是在“走迷宫”：

- 每个岔路口，是一个当前现场状态
- 每条可走的路，是一个合法的钩动作
- 最终出口，是所有业务目标都满足的状态

### 3.2 它不是纯暴力穷举

它不会把所有动作路径都一视同仁地乱试。

它会：

- 提前过滤掉大量明显违法的动作
- 用启发信息优先搜索更有希望的分支
- 在 `beam` 模式下主动裁掉较差分支
- 在 `lns` 模式下先找一个解，再围着差的地方修

所以它是“强约束 + 启发式搜索”，不是无脑枚举。

## 4. 整体流程图：从输入到结果

从系统视角看，完整主流程可以拆成 7 步。

### 4.1 读取主数据

主数据定义了“这个站场世界本身长什么样”。

主要包括：

- 股道
- 台位
- 区域
- 物理分支
- 业务规则

对应代码入口：

- `src/fzed_shunting/domain/master_data.py`

你可以把主数据理解成“地图 + 交通规则”。

### 4.2 归一化输入

把原始 JSON 转成统一、干净、能直接求解的数据结构。

对应代码入口：

- `src/fzed_shunting/io/normalize_input.py`

### 4.3 构造初始状态

把“现在现场长什么样”整理成求解器内部使用的状态对象。

对应代码入口：

- `src/fzed_shunting/verify/replay.py`

### 4.4 生成候选动作

从当前状态里找出所有“下一步可能做、而且合法”的钩动作。

对应代码入口：

- `src/fzed_shunting/solver/move_generator.py`

### 4.5 搜索完整计划

把这些候选动作拼成一条完整计划。

对应代码入口：

- `src/fzed_shunting/solver/astar_solver.py`

### 4.6 回放整条计划

把求出来的钩计划重新一步一步执行，得到每一步快照和最终状态。

对应代码入口：

- `src/fzed_shunting/verify/replay.py`

### 4.7 校验整条计划

独立检查结果是否真的满足业务与路径规则。

对应代码入口：

- `src/fzed_shunting/verify/plan_verifier.py`

一句话总结这 7 步：

> 先把世界和问题描述清楚，再生成合法动作，再搜索，再回放，再校验。

## 5. 先认识几个最核心的术语

如果下面这些词没弄清楚，后面读任何一段都会绕。

### 5.1 股道（track）

车停放或经过的线。

在系统里，股道不仅有名字，还有：

- 类型
- 有效长度
- 能不能停车
- 能不能作为最终目的地
- 连接节点
- 终端分支
- 倒车分支

### 5.2 台位（spot）

比“股道”更细的停放位置。

比如大库内，最终不能只知道“在修 3 库内”，还可能要知道“在 301、302、303 哪个台位”。

### 5.3 区域（area）

比“具体台位”更粗一层的业务目标。

例如：

- 调棚作业位
- 洗南作业位
- 油作业位
- 抛作业位

### 5.4 单钩 / 动作（hook / move）

当前系统里的动作模型非常简单：

- 只有一种动作类型：`PUT`

它表示：

> 把一组连续车辆，从源股道北端挂出来，沿一条完整路径送到目标股道。

一个动作至少包含：

- `sourceTrack`
- `targetTrack`
- `vehicleNos`
- `pathTracks`

### 5.5 block

一组被当成一个整体搬运的连续车辆。

比如：

- `[A]`
- `[A, B]`
- `[A, B, C]`

都可以是 block。

### 5.6 北端前缀

这个词非常重要。

如果一条股道北到南是：

- `A, B, C, D`

那么北端前缀是：

- `A`
- `A, B`
- `A, B, C`
- `A, B, C, D`

不是：

- `B, C`
- `C, D`

因为那些不是当前可直接挂走的最前面一串。

### 5.7 临停线 / 清障

有时候不是目标不能去，而是：

- 前面有挡车
- 路上有挡路股道

这时求解器会考虑先把一些车临时挪到 `TEMPORARY` 类型股道，再继续做真正的目标动作。

### 5.8 状态（state）

状态就是“此刻现场快照”。

这个概念会在全文反复出现。

### 5.9 启发值（heuristic）

不是规则，也不是结果，而是搜索时的一个“方向感”。

它不保证正确性，只帮助搜索器优先看更有希望的分支。

### 5.10 replay

把一条钩计划按顺序重新执行一遍，恢复出每一步状态变化。

### 5.11 verifier

独立检查器，用来确认这条计划不是“看起来能跑”，而是真的合规。

## 6. 输入到底被整理成了什么

求解器不会直接拿原始 JSON 开始搜索，它会先做归一化。

### 6.1 为什么必须归一化

因为原始业务输入里，目标可能是很多种表达方式：

- 直接写股道
- 写大库
- 写作业区
- 写台位编号
- 写“迎检”

这些如果不先统一，后面的搜索逻辑会非常乱。

### 6.2 每辆车归一化后的关键信息

归一化后的车辆，重点有这些字段：

- `current_track`
  - 当前在哪条股道
- `order`
  - 在该股道上的顺序
- `vehicle_length`
  - 车长
- `goal`
  - 最终目标
- `need_weigh`
  - 是否必须先称重
- `is_heavy`
  - 是否重车
- `is_close_door`
  - 是否关门车

### 6.3 目标被统一成 `GoalSpec`

当前目标主要分成 3 类：

#### `TRACK`

意思是：

> 最终必须停在某条明确股道上。

例如：

- 必须到 `存4北`
- 必须到 `机棚`

#### `AREA`

意思是：

> 最终目标是某个区域，只要进入该区域允许的股道/工位即可。

例如：

- 调棚作业
- 洗南作业
- 油作业

#### `SPOT`

意思是：

> 不只是到某条股道，而是必须到某个明确台位。

例如：

- 到 `修3库内` 里的 `304`

### 6.4 为什么“迎检”会影响整个 yard mode

当前系统里有 `yard_mode`：

- `NORMAL`
- `INSPECTION`

当输入里有“迎检”车辆时，会切到 `INSPECTION` 模式。

这会影响一些大库台位的开放范围，例如 `06/07` 台位只在迎检模式开放。

所以不是只改一辆车的目标，而是整个场景的可用台位集合都会变化。

### 6.5 `trackInfo` 有什么用

`trackInfo` 里最关键的是每条股道的可用长度。

后面任何动作只要会让某条股道放不下，就会直接被剪掉。

所以这不是展示信息，而是实打实会参与求解。

## 7. 主数据在这套求解器里扮演什么角色

很多人第一次看求解器时，会把注意力都放在搜索算法上。

但在当前仓库里，主数据的重要性一点不比搜索低。

### 7.1 主数据决定“世界长什么样”

主数据里有：

- 哪些股道存在
- 哪些股道是运行线，哪些是停车线，哪些是临停线
- 哪些股道允许最终停车，哪些不允许
- 哪些分支已经确认可用
- 每个分支多长
- 机车长度按多少算
- 中间路径是否要求净空

所以搜索器不是自己发明规则，而是在主数据定义好的世界里行动。

### 7.2 route oracle 为什么重要

`RouteOracle` 可以理解成“路线专家”。

它负责回答：

- 从源股道到目标股道，拓扑上能不能走
- 应该走哪条更短的路线
- 完整经过哪些 `pathTracks`
- 会经过哪些物理分支
- 总长度是多少
- 有没有用到 `L1`
- 倒车余量够不够
- 路中间有没有被车占住

换句话说：

- 搜索器负责“决定要不要走”
- route oracle 负责“这条路到底能不能走”

### 7.3 为什么同时要有 `branch_codes` 和 `path_tracks`

它们看起来都像“路径”，但用途不同。

#### `branch_codes`

更偏物理分支层面的路径。

主要用来：

- 算总长度
- 检查物理分支状态
- 检查 `L1`
- 检查 reverse clearance

#### `path_tracks`

更偏股道层面的完整经过序列。

主要用来：

- demo 回放
- verifier 检查中间股道净空
- 导出可读结果

可以简单理解成：

- `branch_codes` 更像“走过了哪些道岔/分支”
- `path_tracks` 更像“沿途经过了哪些股道”

## 8. 求解器内部的“状态”到底是什么

这是全文最核心的概念之一。

当前系统里，一个状态主要由 4 部分组成：

- `track_sequences`
- `loco_track_name`
- `weighed_vehicle_nos`
- `spot_assignments`

下面逐个讲。

### 8.1 `track_sequences`

这是最重要的部分。

它记录：

- 每条股道上现在有哪些车
- 顺序是什么

例如：

```text
存2: [A, B]
预修: [C]
修3库内: [D, E, F]
```

这份信息直接决定了：

- 哪些车在最前面
- 哪些 block 当前可挂
- 哪些路径会被中间股道占用卡住

### 8.2 `loco_track_name`

记录机车当前位置。

当前版本里，动作本身不会对机车能力做更复杂建模，但它仍然会在状态里被持续更新，用于：

- 现场快照展示
- workflow 阶段继承
- debug 分析

### 8.3 `weighed_vehicle_nos`

这是“已经完成称重”的车辆集合。

为什么要单独存？

因为“车现在在哪”并不能代表“称没称过”。

例如：

- 某辆车已经离开机库了
- 但它曾经进去过机库

这种信息必须单独记。

### 8.4 `spot_assignments`

这是“已经占了哪个台位”的映射。

它很重要，因为对于大库和作业区来说：

- 不是到股道就结束
- 还要知道有没有占到合法台位

### 8.5 为什么初始状态也要有台位分配

很多人会以为台位只在最后才分。

不是。

当前实现里，如果有些车一开始就已经在需要台位的股道上，系统会在初始状态里就尽量给它们分配台位。

这样做的原因是：

- 后续动作要考虑已有台位是否被占用
- 如果初始就不建这个信息，后面很多约束会失真

## 9. 一个动作（HookAction）具体长什么样

当前系统里的动作结构很朴素：

- `source_track`
- `target_track`
- `vehicle_nos`
- `path_tracks`
- `action_type="PUT"`

可以把一个动作理解成一句完整的话：

> 把 `vehicle_nos` 这组车，从 `source_track` 的北端挂出来，沿 `path_tracks` 经过的完整路径送到 `target_track`。

注意两个点：

### 9.1 `vehicle_nos` 必须是源股道北端前缀

如果不是，动作就会在应用或 replay 时直接报错。

### 9.2 `path_tracks` 不是可有可无的注释

它不是“为了好看顺手带一下”，而是计划的一部分。

因为 verifier 会检查：

- 它是不是从源到目标的完整合法路径
- 中间是不是有挡路股道

## 10. 候选动作到底是怎么生成出来的

这是求解器最关键的一层。

搜索器能不能找到好计划，很大程度上取决于动作生成是否既“够全”又“不过度膨胀”。

下面一步一步讲。

### 10.1 先遍历每条当前有车的股道

对于状态里的每条股道：

- 如果空着，跳过
- 如果有车，尝试从这条股道生成可行动作

### 10.2 再枚举这条股道的所有北端前缀

比如某条股道当前是：

- `A, B, C, D`

系统会依次考虑：

- `[A, B, C, D]`
- `[A, B, C]`
- `[A, B]`
- `[A]`

也就是说，会从长前缀往短前缀试。

### 10.3 每个 block 先过“单钩组合合法性”检查

不是任何一串车都允许作为一个钩动作。

当前代码里至少会检查：

- 纯空车单钩最多 `20` 辆
- 单钩重车最多 `2` 辆
- 重车按 `4` 辆空车折算后，折算总量不能超过 `20`
- 单钩称重车最多 `1` 辆

所以如果一个前缀 block 在这一步就违规，会被直接丢掉。

### 10.4 然后看这组车是不是“目标一致”

求解器不会默认把一串车硬送去同一个地方。

它会先检查这组车的目标是不是一致，包括：

- 目标模式一致
- 目标股道一致
- 允许目标股道集合一致
- 目标区域一致
- 目标台位一致

只有目标一致，系统才会认为“把它们当成一个整体直接送目标”是有意义的。

这背后的直觉是：

- 如果一组车目标不同，一起搬过去通常不是一步到位
- 这类动作会快速制造无效中转

所以“目标一致”本质上是一种搜索空间控制。

### 10.5 如果需要称重，候选目标会被强制改成 `机库`

这是当前逻辑里非常关键的一条。

如果一组车里有人需要称重，而且还没称过，那么候选目标不是最终目标，而是：

- `机库`

这意味着：

- “先称重”不是事后补检查
- 而是在动作生成阶段就被写进了动作候选逻辑

这样做的好处是：

- 搜索树更干净
- 不会生成一堆“看起来快到终点、但其实漏了称重”的伪好解

### 10.6 如果是大库随机目标，会优先把车往更满的库线聚集

对于 `大库:RANDOM`，候选目标股道会排序。

当前策略是：

- 优先选已经更满的库线

为什么这么做？

因为如果随机大库车辆被过度分散，会带来几个问题：

- 台位更碎
- 后续搜索状态更多
- 看起来都合法，但组合更发散

所以这是一种很实用的启发式：

- 尽量聚拢
- 少制造没必要的分散状态

### 10.7 直接送目标不行时，才考虑临停清障

临停不是默认动作，而是补充动作。

当前主要有两类临停请求。

#### A. 前挡车清障

场景是：

- 股道最前面那组车其实已经在自己的合法终点上
- 但它挡住了后面真正应该继续走的车

这时系统会考虑：

- 先把前挡车临时挪走
- 让后面的目标车辆能被挂出来

#### B. 路径清障

场景是：

- 某组目标一致的车，其实已经有明确目标
- 但去目标的中间路径股道被别的车占着

这时系统会预先记录：

- 哪条股道上的车，挡住了谁的目标路径

之后再考虑把这些挡路车送去临停线。

### 10.8 临停线不是乱选的

系统只会从主数据里 `track_type = TEMPORARY` 的股道里找临停目标。

并且还会按距离排序：

- 当前股道到临停线要尽量近
- 临停线再去目标也要尽量近

它会优先选“中转成本小”的临停线。

而且每次只保留很少量临停目标候选，避免搜索树爆炸。

所以当前临停逻辑不是追求全局最优，而是一个**受控的、偏实用主义的清障策略**。

---

## 11. 每个候选动作在进入搜索前还要过哪些硬过滤

动作生成出来不等于能进搜索队列。

在 `_build_candidate_move` 里，它还会继续做一轮“最后把关”。

只要有一条不满足，这个动作就直接丢掉。

### 11.1 目标不能和源相同

这是最基本的。

不允许：

- 从 `预修` 搬到 `预修`

### 11.2 关门车大钩规则不能违规

当前规则是：

- 如果目标不是 `存4北`
- 且单钩车辆数大于 `10`
- 那么关门车不能排在这组车的第一位

这条规则是在动作生成阶段就提前剪枝。

### 11.3 目标股道必须放得下

系统会计算：

- 这组车总长度
- 目标股道当前已占用长度
- 两者相加是否超过 `track_distance`

超了就不允许。

### 11.4 某些作业区有容量上限

例如：

- 调棚 `WORK` 最多 `4`
- 洗南 `WORK` 最多 `3`
- 油 `WORK` 最多 `2`
- 抛 `WORK` 最多 `2`

所以不是路径能走就能进，还要看该区域当前是否已经过载。

### 11.5 台位必须能分出来

这点非常重要。

如果一个动作把车送到某个大库/作业区股道，但那里根本没有可用台位了，那么这个动作会直接失败。

也就是说：

- 求解器不是“先把车送进去，最后再想停哪”
- 而是在动作生成时就预演一次台位分配

只要分不出来，这个动作就不合法。

### 11.6 路径必须真实可达

route oracle 会检查：

- 有无合法 route
- 有无完整 `pathTracks`
- 该 route 对应的 physical branch 是否存在
- 这些 branch 的状态是否已确认

### 11.7 中间股道净空

如果业务规则要求中间路径净空，那么：

- `pathTracks` 的中间那些股道
- 只要被别的车占着
- 这个动作就不允许

这也是为什么“路径清障”会成为需要的动作类型。

### 11.8 `L1 190m` 约束

如果某条路径会经过 `L1` 相关分支，那么列长不能超过 `190m`。

这条规则在 `route_oracle.validate_path` 里检查。

### 11.9 reverse clearance 必须够

当前系统会计算：

- 需要的倒车余量 = 列长 + 机车长度

然后去检查涉及到的 reverse branch 长度是否足够。

如果不够，这个动作也会被判非法。

## 12. 台位分配到底是怎么想的

很多人第一次看大库相关逻辑时，会把“台位分配”想得太晚。

实际上它贯穿了：

- 初始状态
- 候选动作过滤
- 最终目标判定
- verifier 校验

### 12.1 精确台位

如果目标就是具体台位，例如 `304`，那候选台位只有这一个。

只要这个台位不在对应库线、不在当前 yard mode 允许范围内、或者已经被占，就不行。

### 12.2 大库随机台位

如果目标是 `大库:RANDOM`，那候选台位会是：

- 该库线在当前 yard mode 下允许的全部台位

但也不是任意车都能进任意库线。

例如：

- 长车只能进更长的库线

### 12.3 作业区台位

调棚、洗南、油、抛这些作业区也有自己的“工位列表”。

进入这些区域，同样需要分配一个合法工位。

### 12.4 机库称重也被当成一种“台位占用”

对于需要称重的车，进 `机库` 时也会占一个特殊工位：

- `机库:WEIGH`

这样系统可以统一地处理“有没有位置”“是不是已占用”等问题。

## 13. 一个动作被接受之后，状态怎么变化

当搜索器决定“这个动作可以走”，它会调用 `_apply_move` 把状态往前推进一步。

这个过程非常机械，但很重要。

### 13.1 检查源股道前缀是否匹配

系统会先确认：

- `move.vehicle_nos`
- 是否真的等于 `source_track` 当前最北端那一串车

如果不是，直接报错。

### 13.2 从源股道删除这组车

也就是把前缀切掉。

### 13.3 追加到目标股道尾部

注意是追加到尾部。

这意味着：

- 车到了目标股道后，会排在原来已有车辆的后面

所以顺序变化一直是动态累积的。

### 13.4 清掉旧台位，再分配新台位

这组车离开原股道时：

- 旧台位占用要取消

进入新股道时：

- 如果目标需要台位，就重新分配

### 13.5 如果目标是 `机库`，就更新已称重集合

系统会把这些车记进：

- `weighed_vehicle_nos`

### 13.6 更新机车位置

机车位置会更新到目标股道。

## 14. 搜索主循环到底是怎么工作的

现在进入最核心的求解主循环。

你可以把它理解成一个不断重复的过程：

1. 从“最值得先看”的状态开始
2. 检查它是不是已经完成
3. 如果没完成，就展开下一层动作
4. 把新状态放回等待队列
5. 重复

### 14.1 队列里存的是什么

每个排队项里至少有：

- 当前状态
- 到达该状态的计划
- 这个状态的优先级
- 状态键

也就是说，搜索器并不是只记“当前状态”，还记“我是怎么走到这里的”。

### 14.2 当前主成本是什么

当前求解器把：

- `cost = 已经用了多少钩`

作为主成本。

这代表它天然更偏向：

- 钩数更少的方案

### 14.3 当前启发值是什么

启发值非常简单：

- 还有多少辆车当前不在允许目标股道里

注意它只回答一件事：

> 看起来还有多少车连“大方向”都没到位？

它**没有**完整考虑下面这些事：

- 称重是否完成
- 精确台位是否就位
- 关门车顺位是否已经满足

为什么还敢这么做？

因为它的定位就是：

- 便宜
- 粗
- 只提供方向感

真正严谨的正确性仍然靠目标判定和 verifier。

### 14.4 优先级怎么计算

#### `exact`

优先级主要按：

- `cost + heuristic`

#### `weighted`

优先级会变成：

- `cost + heuristic_weight * heuristic`

也就是让搜索器更看重“离终点近不近”。

#### `beam`

和 `exact` 的打分接近，但会额外做宽度截断。

### 14.5 主循环每轮做什么

每轮大致是：

1. 从优先队列里拿出分数最好的状态
2. 如果它已经不是当前最优版本，就跳过
3. 检查是否已达成目标
4. 如果没达成，就生成候选动作
5. 对每个动作应用得到新状态
6. 算新状态代价和优先级
7. 放回队列

### 14.6 为什么会有 `best_cost`

因为同一个状态可能通过不同路径被走到。

如果：

- 某个状态之前已经用更少钩数到达过

那现在这个更差的版本就没有保留价值。

所以 `best_cost` 的作用是：

- 记录“到达某个状态键的当前最小成本”
- 避免重复扩展更差版本

---

## 15. 什么叫“目标状态已经达成”

求解器不会只看“是不是到了目标股道”。

当前 `_is_goal` 的判断要比这严格得多。

### 15.1 必须在允许的最终股道里

如果某辆车当前不在 `allowed_target_tracks` 里，直接不算完成。

### 15.2 需要称重的车必须已经称重

只要：

- `need_weigh = True`
- 但车不在 `weighed_vehicle_nos` 里

就不算完成。

### 15.3 精确台位必须完全匹配

如果目标是 `SPOT`：

- 当前台位必须 exactly 等于目标台位

### 15.4 大库随机和作业区目标也必须有合法台位

即使车已经到了正确库线/作业区股道，如果还没有合法台位分配，也不算完成。

### 15.5 关门车在 `存4北` 不能排前 3

这也是目标判定的一部分。

也就是说，求解器把这条业务规则视为“终局合法性”的组成部分，而不是展示层附加说明。

---

## 16. 为什么要做状态去重，以及它具体怎么去重

如果不去重，搜索会非常容易失控。

因为经常会出现这种情况：

- 你先把 A 临停，再把 B 挪动
- 或者先把 B 挪动，再把 A 临停
- 最后现场居然变成一样

如果每次都把这种重复局面重新搜一遍，代价会非常大。

### 16.1 当前 `state_key` 包含什么

当前状态键主要包含：

- 所有非空股道的车辆序列
- 已称重车辆集合
- 台位分配

这三类信息加起来，基本就能唯一刻画“当前局面”。

### 16.2 为什么随机大库台位会做一定程度的忽略

如果目标只是“大库随机停放”，并没有指定精确台位，那么：

- `101` 和 `102` 的差别，对很多搜索判断来说并不关键

因此系统会做一个小优化：

- 对这类随机大库车辆，状态键里忽略纯数字台位差异

这样能减少很多“业务上差不多，但状态字符串不同”的重复分支。

换句话说，它是在说：

> 对当前搜索来说，这两种局面可以当作同一种看。

## 17. `beam` 为什么会快，代价又是什么

`beam` 的核心不是换了一套完全不同的动作逻辑，而是：

> 每轮扩展后，只保留当前最好的前 `beam_width` 个状态。

### 17.1 它为什么快

因为普通 best-first 搜索会保留很多候选分支。

而 `beam` 主动告诉系统：

- 不要都留着
- 只保留最有希望的一小撮

这能大幅减少：

- 队列大小
- 后续展开次数
- 内存与时间开销

### 17.2 它为什么可能错过更好解

因为某些真正好解，在早期看起来可能并不出色。

如果 `beam_width` 太小：

- 这些“先吃点亏、后面很优秀”的分支
- 可能会被提前裁掉

所以 `beam` 的本质是：

- 用部分最优性风险
- 换取更强的速度和规模能力

### 17.3 当前 `beam` 还会做后修补

当前实现不是“找到一个 beam 解就立刻结束”，而是还会再做几轮局部修补，看看能不能把钩数继续压短。

## 18. `lns` 到底是在干什么

`lns` 比 `beam` 再走远一步。

它的思路不是：

> 从头彻底重算一遍。

而是：

> 先拿到一个还不错的解，再围着差的地方局部重算。

### 18.1 第一步：先拿初解

当前实现里：

- 如果给了 `beam_width`，先用 `beam`
- 否则先用 `weighted`

先求一条可用计划出来。

### 18.2 第二步：把这条计划 replay 成一串中间快照

这样系统就知道：

- 第 1 钩之后现场是什么样
- 第 2 钩之后是什么样
- …

### 18.3 第三步：挑几个最值得下刀的“切点”

当前会优先关注：

- 反复被触碰的热点股道
- 涉及临停线的动作
- 涉及 `存4南` 的动作

因为这些地方通常最容易说明：

- 当前计划里有来回腾挪
- 有比较重的清障痕迹
- 可能存在压缩空间

### 18.4 第四步：从切点后面重新算后缀

它会保留前半段计划不动，然后：

- 从某个中间快照开始
- 重新构造一个“局部问题”
- 再求一段新的后缀计划

### 18.5 第五步：已经满足目标的车会被冻结

这一步很关键。

当前实现里，如果某辆车在切点状态已经满足：

- 最终股道对了
- 该称重也称重了
- 该有台位也有台位了

那么它的目标会被改写成：

- 保持当前状态不动

这相当于告诉 repair 阶段：

- 这些车不用再折腾
- 你重点修剩下那部分

### 18.6 什么叫“更好”

当前比较顺序是：

1. 钩数更少
2. 分支数更少
3. 路径总长度更短

所以 `lns` 的目标不是抽象的“更优”，而是一个非常明确的多级比较标准。

## 19. replay 为什么是独立的一层

很多人会问：

> 搜索时不是已经应用过动作了吗？为什么还要 replay 一遍？

答案是：

- 搜索里的 `_apply_move` 是内部状态推进
- replay 是针对“导出的钩计划”重新逐钩执行

这两层分开非常重要。

### 19.1 搜索内部推进，主要是为了继续算

它关注的是：

- 当前局面怎么变成下一局面

### 19.2 replay 更像“把结果重新演一遍”

它关注的是：

- 导出的钩计划是否自洽
- 每一步快照是什么
- demo 要展示什么
- verifier 用哪个前置状态检查每一钩

### 19.3 replay 的产物不仅是最终状态

它还会生成一串 `snapshots`：

- 初始状态
- 第 1 钩后
- 第 2 钩后
- …
- 最终状态

这正是 UI 能做逐步回放的基础。

## 20. verifier 为什么必须存在

当前仓库有一个非常重要的设计原则：

> 求解结果不以“求解器自己说可以”为准，而要以独立 verifier 复验为准。

这非常关键。

因为搜索器就算已经很严格，也仍然可能出现：

- 某些边角规则漏掉
- 导出计划字段不完整
- 某步动作在 replay 口径下不自洽
- 某些状态内部看着像完成，外部口径却不合规

### 20.1 verifier 会先尝试 replay

如果连 replay 都过不去，例如：

- `vehicleNos` 不是源股道北端前缀
- 台位分配失败

那么这条计划就直接失败。

### 20.2 verifier 会做全局校验

包括：

- 每辆车最终是不是在允许股道
- 该称重的是不是称了
- 精确台位是否匹配
- 大库/作业区台位是否合法
- 关门车顺位是否合法
- 最终股道是否容量超限
- 是否停到了不允许停车的运行线

### 20.3 verifier 还会做逐钩校验

每一钩都会检查：

- `pathTracks` 是否存在
- 路径首尾是否匹配
- 是否引用了未知车辆
- 单钩车辆组合是否合法
- route/path 是否真的成立
- 中间股道是否被占用
- reverse clearance 是否足够
- 关门车大钩规则是否违规

所以 verifier 的存在，保证了：

- 不是“好像算出来了”
- 而是“真的能按统一口径复验通过”

## 21. 多阶段 workflow 为什么不是另一套求解器

当前仓库既支持单阶段，也支持多阶段 workflow。

它们底层不是两套逻辑。

### 21.1 workflow 的本质

workflow 做的事情很简单：

1. 先解第 1 阶段
2. 拿第 1 阶段的最终状态当第 2 阶段的初始状态
3. 再解第 2 阶段
4. 如此继续

### 21.2 阶段间继承什么

当前会继承：

- `track_sequences`
- `loco_track_name`
- `weighed_vehicle_nos`
- `spot_assignments`

也就是说，第二阶段不是凭空开始，而是建立在第一阶段结束后的真实现场上。

### 21.3 为什么这点很重要

这意味着多阶段能力不是“每阶段都重新想象一个干净世界”，而是：

- 真正把前阶段留下的现场包袱带到后阶段

这更符合实际业务。

## 22. 用一个非常小的例子把全过程串起来

下面用一个极简例子，把整条思路走一遍。

假设当前有 3 条线：

- `存2`
- `临1`
- `机库`
- `修3库内`

当前现场是：

- `存2: [A, B, C]`

目标是：

- `A` 最终其实还留在 `存2`
- `B` 必须先去 `机库` 称重，再去 `修3库内`
- `C` 最终去 `修3库内`

### 22.1 初始状态怎么看

求解器会看到：

- `A` 在最前面
- `B` 被 `A` 挡住
- `C` 被 `A` 和 `B` 一起挡住

### 22.2 当前能尝试哪些 block

只可能是：

- `[A]`
- `[A, B]`
- `[A, B, C]`

不可能直接拿 `[B]`。

### 22.3 先看直接送目标是否合理

- `[A]` 虽然能拿，但它其实已经在自己的合法目标上
- `[A, B]` 目标不一致
- `[A, B, C]` 目标也不一致

于是“直接一步送终点”的动作就不太理想。

### 22.4 前挡车清障触发

系统发现：

- `A` 已经在合法位置
- 但它挡住了后面的 `B`

于是会考虑：

- 先把 `[A]` 送去某条临停线，比如 `临1`

### 22.5 新状态出现

这时现场变成：

- `存2: [B, C]`
- `临1: [A]`

### 22.6 现在 `B` 到了最前面

由于 `B` 需要称重，系统不会直接把它送 `修3库内`，而是会先生成：

- `存2 -> 机库`

### 22.7 `B` 称重后，再看后续

这时状态里会记录：

- `B` 已经称过重

后面就可以合法地继续往 `修3库内` 送。

### 22.8 `C` 也逐步送入目标

### 22.9 最后把 `A` 从临停线送回合法终点

于是，一条完整计划可能会长成：

1. `A: 存2 -> 临1`
2. `B: 存2 -> 机库`
3. `B: 机库 -> 修3库内`
4. `C: 存2 -> 修3库内`
5. `A: 临1 -> 存2`

这个例子说明了一件最核心的事：

> 求解器经常不是直接搬“目标车”，而是先处理挡路车和前置条件。

这也是它和“静态分配器”最大的区别。

## 23. 如果求解失败，通常意味着什么

当你看到：

- `No solution found`

不应该第一反应就是“算法坏了”。

更常见的可能是下面几类原因。

### 23.1 当前场景真的无解

例如：

- 目标股道根本不允许最终停车
- 台位数量不够
- 长车没有可去的大库线
- 路径物理上不连通

### 23.2 有目标，但前置条件永远达不成

例如：

- 需要称重，但机库路径被永远卡死

### 23.3 路径清障空间不足

比如：

- 临停线太少
- 关键挡路股道上的车没有合法暂存去处

### 23.4 搜索策略太保守

尤其在：

- `beam_width` 太小

时，可能不是问题无解，而是：

- 真正的好分支被过早裁掉了

### 23.5 输入合同有问题

例如：

- 顺序重复
- 股道名不存在
- 目标写法不合法

这些在归一化阶段就可能直接报错。

## 24. debug 统计信息可以怎么读

当前求解器支持记录一批 debug stats。

它们能帮助你判断：

- 当前难点在哪
- 搜索是不是分叉太多
- 临停动作是不是很多
- 哪些股道被频繁触碰

### 24.1 `expanded_states`

真正展开过多少状态。

如果很大，说明搜索树比较宽或比较深。

### 24.2 `generated_nodes`

总共生成过多少候选状态。

一般会比 `expanded_states` 更大。

### 24.3 `candidate_moves_total`

动作生成阶段总共产生了多少候选动作。

如果这个值很夸张，通常说明：

- 某些状态分叉太多

### 24.4 `candidate_direct_moves` / `candidate_staging_moves`

这两个值能帮你看出：

- 当前解主要靠直接送目标在推进
- 还是靠大量临停清障在推进

如果 `staging_moves` 很高，通常说明现场比较堵。

### 24.5 `moves_by_target`

看哪些目标股道被频繁尝试。

如果某个临停线被高频使用，通常说明它是关键缓冲区。

### 24.6 `top_expansions`

这个信息很有用。

它会记录那些“分叉最多”的状态展开点。

你可以把它理解成：

- 求解器最纠结的几个现场瞬间

## 25. 当前四种 solver 模式，应该怎么理解取舍

最简单的理解方式如下。

### `exact`

适合：

- 小场景
- 想完整看搜索行为
- 不想太早裁剪分支

特点：

- 更稳
- 更慢

### `weighted`

适合：

- 想保留搜索框架
- 但希望更快朝“像是终点”的方向推进

特点：

- 更贪心
- 可能更快

### `beam`

适合：

- 场景更大
- 更关心速度
- 能接受一定最优性风险

特点：

- 很实用
- 但可能错过更好分支

### `lns`

适合：

- 先有一个可用解
- 再希望在热点局部继续压缩

特点：

- 更像“先求出来，再打磨”

## 26. 当前这套求解器“是什么”，又“不是什么”

这个认识很重要，不然容易期待错方向。

### 26.1 它是什么

它是一个：

- 强约束
- 可解释
- 可回放
- 可复验
- 以实际业务规则为中心

的调车规划内核。

### 26.2 它不是什么

它不是：

- 连续动力学仿真器
- 完整机车作业时刻仿真器
- 保证全局最优的数学规划器
- 完全无启发式的纯精确求解器

特别是临停清障与 `lns` repair，目前都更偏：

- 实用主义
- 局部优化
- 先求能用，再逐步改强

## 28. 最后再用一句话收口

如果你已经读到这里，可以把当前求解器总结成下面这句话：

> 它先把调车问题翻译成“状态 + 合法单钩动作”，再在强业务约束下搜索一条完整计划，最后通过 replay 和 verifier 保证结果不只是算出来了，而且真的能被复盘和复验。

如果要再口语一点，就是：

> 它不是直接给你一个神秘答案，而是在一个有地图、有规则、有挡路关系的站场里，一步一步找到“下一钩该怎么干”，最后把这些下一钩串成整条计划。
