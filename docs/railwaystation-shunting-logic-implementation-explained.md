# RailwayStation 调车逻辑实现深度解析

## 1. 这份文档讲什么

这份文档不是重新复述《福州东车辆段业务说明》或《福州东调车业务说明》，而是专门解释当前 `RailwayStation` 仓库里那套 **C# 主实现** 到底是怎么工作的。

更具体地说，它回答 4 个问题：

1. 这个项目到底在解什么问题。
2. 代码里是怎样把“股道、车辆、机车、目标顺序”变成一个可以运行的状态机的。
3. 主算法每一轮到底在做什么，为什么会先跑任务、再选车、再缓存、再放车。
4. 这套实现和新版福州东业务文档之间，哪些地方是一致的，哪些地方还有明显差距。

如果你刚接手项目，建议把这份文档和下面几类文件配合着看：

- `README.md`

- `福州东车辆段业务说明.md`

## 2. 先用一句人话概括这套系统

这套系统要做的事，可以用一句话概括：

> 已知一批车现在分别停在哪些股道、最终要去哪条股道并且按什么顺序摆好，系统自动生成一串 `Get / Put / Weigh` 调车动作，把它们按规则搬过去。

这里最容易误解的一点是：

- 它不是一个“求最短路径”的图算法演示。
- 它也不是一个“把所有动作一次性全局最优求出来”的数学规划器。
- 它更像一套 **规则驱动、任务驱动、带启发式缓存选择的调度引擎**。

所以你看到的代码重点不是：

- 建一个超大的状态图然后求最优解，

而是：

- 不断判断当前局面；
- 看有没有必须先做的任务；
- 如果没有，再决定“下一辆最值得取的车是谁”；
- 遇到挡车、容量不够、目标线没清空、机走线要特殊处理时，再触发对应的任务或缓存动作。

这个仓库里同时有几类东西：

## 4. 当前实现到底在解决什么“抽象问题”

如果把业务词都拿掉，当前实现抽象出来其实就是一个“多栈重排问题”：

- 每条股道上当前有一串车，顺序不能乱跳。
- 每条目标股道上未来也有一串目标车，顺序必须满足。
- 调车机不是纯运输工具，而是一条特殊的中间栈。
- 某些线可以暂时当缓存线，但有容量、可达性和业务副作用。
- 目标车如果被别的车压住，就得先把上面的车挪走。
- 如果机车挂得太多、重车太多、称重车还没处理，算法就必须中途绕一下。

因此，这个问题最核心的不是“从 A 点到 B 点怎么走”，而是：

> 怎么在不破坏目标编组顺序的前提下，用最少的折腾，把一串串车从当前栈搬成目标栈。

这也是为什么代码里大量逻辑都围绕下面这些概念展开：

- 栈顶车
- 连续车组
- 当前线是否清空
- 目标线是否清空
- 某条线是不是缓存线
- 机车上这批车能不能直接放到目标线

## 5. 输入数据是怎么变成内存状态的

### 5.1 输入入口

离线入口在 `RailwayStation.Console/Program.cs`。

它会先读：

- `Data/map.xlsx`
- 一批场景 Excel，例如 `2025-09-04-noon.xlsx`

在线入口在 `RailwayStation.WebApi/EndPoint/PlanEndPoint.cs`。

它会接收：

- `StartStatus`
- `EndStatus`

然后同样转成一批 `TrackLine` 和 `Car` 对象，再交给 `BackwardConstructionAlgorithm`。

### 5.2 `map.xlsx` 提供什么

`DataProvider.GetMapInfo()` 会从 `map.xlsx` 里读取两类主数据：

1. `DistanceMatrix`
2. `Capacity`

这两类数据的作用完全不同：

- `Capacity` 用来约束一条线还能不能放得下更多车。
- `DistanceMatrix` 在当前实现里主要用于 **缓存线选择启发式**，不是完整路径规划。

这一点后面很重要，因为它说明：

> 当前代码虽然读了“距离矩阵”，但并没有把“完整走行路径”作为主要输出对象。

### 5.3 当前状态和目标状态怎么读

`DataProvider.InitCars()` 会读场景 Excel 里的两个 sheet：

- `start`
- `end`

然后：

- `start` 里的每一行生成一辆 `Car`，放进对应股道的 `CurrentList`
- `end` 里的每一行把同一辆车挂到目标股道的 `TargetList` 和 `OriginTargetList`

这里有两个关键点：

1. 当前线和目标线都不只是“在哪条线”，还带着顺位。
2. 目标状态会同时维护两份列表：
   - `TargetList`：剩余还没取到手的目标序列
   - `OriginTargetList`：原始完整目标序列，后续用于判断最终顺序是否正确

### 5.4 线名归一化

`DataProvider.GetCarTrackLine()` 会把 Excel 里的全称做一层归一化，比如：

- `修1库内` 这类名会被拆成主线名 + 补充后缀
- `存5北/存5南`
- `喷漆... / 调梁... / 洗罐... / 机走...`

这说明当前实现里的线路模型比较“老”也比较“粗”：

- 主线名只保留一层抽象；
- 更细的业务位置信息，很多都塞在 `TargetLineName_Second` 里。

所以如果你拿这套代码去对照新版福州东文档，会感觉它还没有完全升级到“区域 / 台位 / targetMode”那一套更精细的表达。

## 6. 这套实现最重要的 5 个对象

### 6.1 `TrackLine`：一条股道，不只是名字和容量

`TrackLine` 是整个系统的核心状态容器之一。

它同时维护三组列表：

- `CurrentList`：现在这条线上的车
- `TargetList`：这条线还没完成的目标车序列
- `OriginTargetList`：原始完整目标车序列

它还维护很多关键状态：

- `IsCanArrived`：当前能不能取到 / 放到这条线
- `IsCache`：当前能不能把它当缓存线
- `IsCleared`：线上还需移动的车是否已经清空
- `IsFinished`：线上当前车辆是否已经达到目标完成态
- `Rem_capacity`：剩余可用长度

最重要的是，`TrackLine` 还带事件：

- `OnCurrentCleared`
- `OnTargetCleared`
- `OnBlocked`
- `OnFinished`

所以它不是被动数据结构，而是会主动通知管理器和算法：

- 某条线清空了
- 某条线目标取完了
- 某条线被占住了
- 某条线已经完成了

当前调车逻辑里很多“突然创建一个任务”的行为，本质上都是这些事件触发出来的。

### 6.2 `Car`：不仅有当前位置和目标位置，还有一堆推导属性

`Car` 的基础字段很直观：

- `CurrentLine`
- `TargetLine`
- `OriginLinePosition`
- `TargetLinePosition`
- `IsHeavy`
- `IsWeigh`
- `IsWeighed`
- `Length`

但真正关键的是它的推导属性：

- `IsNeedMove`
- `IsCurrentTop`
- `IsTargetTop`
- `CurrentDepth`
- `ContinuousCars`
- `IsTargetLineWantedContinuous`
- `NextTargetCar`

这几个属性合在一起，决定了算法会不会优先处理它。

### 6.3 `Train`：调车机被建模成一条特殊的“动态缓存线”

当前实现最有辨识度的一点是：

> 调车机在代码里直接继承自 `TrackLine`。

也就是说，机车不是一个外部搬运器，而是被看成一条特殊股道 `train`。

这带来两个后果：

1. 机车上的车组顺序和普通股道上的车序一样，可以直接用同一套栈逻辑处理。
2. 算法在很多时候会把机车当“临时中转区”甚至“短期缓存区”来思考，而不是只把它看成运输动作的执行者。

`Train` 额外提供了几个关键判断：

- `WantedCar`：机车当前最想接上的下一辆车
- `IsContainAllLineTarget()`：机车上是否已经包含某条目标线的全部剩余目标
- `GetNotContainTargetCars()`：某条目标线还缺哪些车

### 6.4 `Operation`：输出不是路径，而是动作记录

当前主输出对象是 `Operation`。

它记录的不是“完整物理路径”，而是调度动作：

- `Get`
- `Put`
- `Weigh`

每条 `Operation` 会带上：

- 动作发生在哪条线
- 这次搬了哪些车
- 操作后的机车状态
- 线路前后状态

所以从当前系统角度看，输出更像：

> 结构化的调车作业日志

而不是：

> 带完整路径节点和道岔序列的作业计划书

### 6.5 `TaskItem` / `TaskManager`：第二套大脑

如果只看 `BackwardConstructionAlgorithm.Run()`，你会以为算法是：

- 选一辆车
- 搬一下
- 再选一辆车

但实际上真正支配流程的是另一层：

- `TaskItem`
- `TaskManager`
- `BCTaskManager`

任务有：

- ID
- 优先级
- 依赖
- 状态
- 是否允许跳过

状态包括：

- `NotStarted`
- `Running`
- `Completed`
- `Skipped`
- `Paused`

这使得算法不是单线流程，而是：

> 每一轮先看看当前有哪些任务已经该做，做完了再决定是否允许继续普通取车。

## 7. 一定要讲清楚的核心心智模型：三条线和一次反转

如果你只记住这篇文档一件事，我建议你记这个：

> 当前实现本质上是在管理“当前股道栈 -> 机车栈 -> 目标股道栈”的顺序反转关系。

### 7.1 为什么顺序会“反”

原因在于当前代码里很多操作都是按“栈顶插入 / 栈顶弹出”做的：

- 从股道取车：取的是 `CurrentList[0]`
- 放到机车：插到机车 `CurrentList[0]`
- 从机车放车：取的是机车 `CurrentList[0]`
- 放到目标线：插到目标线 `CurrentList[0]`

所以会发生一个典型现象：

假设某目标线最终想要的顺序是：

- A
- B
- C

如果算法从某条当前线依次把 `A -> B -> C` 取到机车上，那么机车上的顺序会变成：

- C
- B
- A

接着再从机车依次把它们放回目标线，目标线上就会恢复成：

- A
- B
- C

这就是“反推式构造”在当前代码里的真实含义：

- 取车阶段看起来是正着取，
- 机车中间会反一次，
- 放车阶段再反一次，
- 最终目标线顺序才会正确。

### 7.2 为什么代码老盯着“连续车组”

因为如果一批车在当前线里本来就是连续的，而且它们去的是同一条目标线、顺序也正好接得上，那么一次成组取走，收益最大：

- 少拆散
- 少缓存
- 少钩
- 后续更容易直接放车

这就是 `Car.ContinuousCars` 的作用。

### 7.3 `IsTargetLineWantedContinuous` 到底在判断什么

这个属性很关键，但第一次读代码最容易看懵。

它判断的是：

> 以当前这辆车为起点形成的连续车组，是否正好等于“目标线当前还需要的那一段”，并且顺序对得上。

在线路上和在机车上，这个判断方向是不一样的：

- 在线路上时，连续目标通常按目标顺序正向增长。
- 在机车上时，因为刚刚经历过一次反转，所以要按目标尾部的逆序去匹配。

这就是为什么当前实现可以做到：

- 一边取车，一边从 `TargetList` 里剥掉“已经拿到手的那部分”；
- 等这批车在机车上堆好后，再判断能不能整段直接放回目标线。

## 8. 算法初始化阶段到底做了什么

`BackwardConstructionAlgorithm` 的构造函数和 `Run()` 前半段，主要完成 6 件事。

### 8.1 对每条线的当前车和目标车做排序

`_SortTrackLineCars()` 会把：

- `CurrentList` 按 `OriginLinePosition`
- `TargetList` 按 `TargetLinePosition`
- `OriginTargetList` 按 `TargetLinePosition`

统一排好。

这样后面所有“栈顶”“深度”“连续车组”的判断才有意义。

### 8.2 创建管理器

会创建：

- `TrackLineManager`
- `CarManager`
- `OperationManager`
- `BCTaskManager`

它们各自负责：

- 线路可达性与缓存策略
- 剩余车辆和称重车辆统计
- 输出动作记录
- 任务调度

### 8.3 创建一条特殊的 `train` 线

这里是整个设计最关键的一步。

算法会新建一个 `Train`，名字叫 `train`，并把它塞进 `TrackLines` 字典里。

从这一步开始，调车机就正式变成了和普通股道同构的状态容器。

### 8.4 订阅线路事件

对每条线都会挂上事件处理器：

- 当前线清空
- 目标清空
- 线路被阻塞
- 线路完成

后面很多自动行为，例如：

- 某条线清空了就开放别的线
- 某条线目标取完了就创建 `Put_xxx`
- 机走线堵住后喷漆和洗罐关闭

都来自这里。

### 8.5 初始化线路可达性和缓存属性

`TrackLineManager` 在初始化时会：

- 给部分线设置默认不可达
- 标注哪些线能当缓存线
- 绑定缓存策略 `CacheStrategy_simple`

当前代码里，修线、卸轮、机走这些线会有比较明显的特殊处理。

### 8.6 创建首批任务

`_InitTask()` 会先创建：

- `Clear_机走`

然后创建一批 `Get_xxx` 任务，重点针对：

- 喷漆
- 洗罐
- 抛丸线
- 机库线
- 调梁
- 老预修
- 存1线
- 存2线
- 存3线

如果还有未称重车辆，还会再加一个：

- `Weigh`

这一步透露了当前实现的一个很强的工程偏见：

> 它不是完全中性的“谁都一样看”，而是从初始化开始就给一批重点线路预埋了取车优先级。

## 9. 主循环怎么运行

`Run()` 的主循环可以翻译成人话：

1. 还有车没到目标吗？
2. 如果有，先看当前有哪些任务已经准备好了。
3. 先跑任务。
4. 只要任务引起了状态变化，就立刻进入下一轮，不急着取普通车。
5. 如果任务这轮没推进，再从全场找“最佳目标车”。
6. 把这辆车及其连续车组尽量取到机车上。
7. 记录操作。
8. 继续下一轮。

这意味着系统不是“任务流”和“取车流”两条独立逻辑，而是：

> 任务优先，普通取车其次。

这样设计的原因很现实：

- 某条目标线刚清空，应该马上放车；
- 某辆称重车条件成熟，应该先称；
- 机走线有车后，某些线必须关闭；
- 某条缓存线暂时成了死胡同，得先清出来；

如果不让这些高优先逻辑抢占，算法就会走出很多低质量动作。

## 10. 任务系统在当前实现里分别干什么

### 10.1 `GetCar`：不是“立刻取车”，而是“提升某条线被选中的概率”

`GetCar.Execute()` 做的事很简单：

- 如果这条目标线已经没剩余目标，任务完成。
- 否则提高这条线的 `Priority`。

所以它更像一个“偏向器”：

> 让主循环后续在 `_GetBestTarget()` 里更容易挑中这条线的目标车。

### 10.2 `ClearLine`：本质上是“让挡事的线尽快被处理”

`ClearLine` 的目标不是一定立即暴力清空，而是推动这条线尽快被清掉。

它会先判断：

- 当前线栈顶车前面的目标链条是否可达；

如果可达，它就提升这条线里所有相关目标线的优先级，让系统尽快把这些车取走。

所以大多数情况下，`ClearLine` 的真实作用是：

> 发出一个“这条线该尽快清了”的信号。

### 10.3 `PutCar`：把机车上已经凑齐的一批车放回目标线

这是当前实现里最常用的放车任务。

它先判断这件事是不是该做：

- 机车上有没有车
- 目标线可不可达
- 目标线清没清空
- 机车车头上有没有一辆更应该先去取的“WantedCar”
- 机车上是否已经包含这条目标线的全部剩余目标

如果这些条件不满足，它要么跳过，要么创建额外任务。

如果满足，它会做两步：

1. 先把车头那些“不是这个目标线的车”缓存到别的线。
2. 再把车头上属于这个目标线、而且顺序已经对上的连续车组放进去。

所以 `PutCar` 的本质是：

> 为了把一条目标线一次放顺，它会先主动把机车上不相关的车临时甩开。

### 10.4 `Weigh`：称重不是主流程，而是一个抢占式插入步骤

称重任务的处理逻辑是：

- 先找机车上还没称过的称重车；
- 只有当这辆称重车后续相关车已经基本都到位时，才允许称；
- 机库线没清空也不能称；
- 如果称重车前面还有别的车，就先把别的车缓存走；
- 然后产生一个 `Weigh` 操作，把这辆车标记成 `IsWeighed=true`。

这里可以看出当前实现对称重的理解是：

- 称重不是目的地；
- 称重是过程中的插入动作；
- 它会打断正常的“取车 -> 放车”节奏。

### 10.5 `PutCarToJZLine`：最复杂、最像“总调度开关”的任务

这就是整个系统最难读的一块。

它处理的不是普通目标线，而是特殊的 `机走` 体系。

它做的事包括：

- 决定什么时候把去修线/卸轮线等的车先集中到机走线；
- 当机走线开始工作后，暂停很多别的任务；
- 提升去机走体系相关目标的优先级；
- 在“放到机走”和“继续缓存到其他线”之间来回切换；
- 条件成熟后开放修理线；
- 存4完成后再把机走线上的车整体取回机车继续处理。

可以把它理解成：

> 当前实现专门为“修线 / 卸轮 / 机走预修”这一类复杂业务，额外造出来的一个阶段控制器。

## 11. 系统如何挑出“下一辆最值得处理的车”

`_GetBestTarget()` 是主循环里最关键的决策函数。

它不是穷举最优，而是启发式选择。

### 11.1 第一优先：能不能顺着机车当前这串车继续接

如果 `train.WantedCar` 存在，而且那辆车当前就在某条线的栈顶、可直接取到，那就直接选它。

这说明系统非常重视：

> 已经在机车上形成的那条目标链条，能不能继续往后接。

这样做的收益很明显：

- 少换目标线
- 少拆机车上已有的连续车组
- 更容易后续整段放车

### 11.2 第二优先：能直接取到的目标车

如果某条目标线的当前目标车本来就在所在股道栈顶，而且当前线可达，那么它比埋得深的车更优先。

### 11.3 第三优先：目标线优先级

前面提到的 `GetCar`、`ClearLine`、机走相关逻辑都会改变目标线 `Priority`。

所以 `_GetBestTarget()` 实际上是在吃“任务系统喂给它的偏好”。

### 11.4 第四优先：埋得浅

如果都不是栈顶，那就优先挑埋得浅的车，也就是 `CurrentDepth` 小的车。

因为这种车要挪开的挡车少，代价更低。

### 11.5 第五优先：目标线剩余目标多

目标线 `TargetList.Count` 越大，说明后续潜在可连续收益越大。

这本质上是一种“批量收益偏好”：

> 先干那些后续可能一口气连着干更多的线。

所以 `_GetBestTarget()` 的本质不是“求最优”，而是：

> 在当前局面下，优先挑一个最可能带来连续收益、少缓存、少换线的目标。

## 12. 真正的取车动作是怎么发生的

### 12.1 简单情况：目标车就在栈顶

如果目标车就是当前线栈顶，而且机车容量够，那么算法会直接调用：

- `Actions.MoveCarFromLineToTrainContinuously()`

这个方法会：

1. 先把目标车从当前线弹出，压到机车栈顶；
2. 如果后面的车和它构成连续目标车组，而且机车还没超限，就继续一口气拉上来。

这就是当前实现里“一钩尽量多拉一点”的代码体现。

### 12.2 复杂情况：目标车被挡住了

如果目标车不在栈顶，算法会进入 `_MoveBlockCarsToCache()`。

它的逻辑是：

1. 先把压在目标车前面的连续车组取到机车；
2. 再把这些挡车整体放到一条缓存线；
3. 然后目标车才暴露出来；
4. 最后再去取真正目标车及其连续车组。

这里有个非常重要的设计选择：

> 挡车被挪开后，很多时候不会“立刻原路送回”，而是直接留在缓存线，等后续按正常流程再处理。

这说明当前实现不是把缓存看成一次局部事务，而是把缓存看成全局状态的一部分。

### 12.3 容量不够怎么办

如果机车再取这批车就会超限，算法会分情况处理：

- 如果还没有 `Put_机走` 任务，就先创建机走放车任务；
- 如果已有机走任务，而且当前要取的是连续车组，就尝试先缓存机车上部分车；
- 极端分支里还有一些 `throw new Exception("目前没有遇到这样的数据，还未实现")` 的兜底。

这说明当前实现对“机车超限”的处理是：

- 有一套主路径；
- 但并没有把所有边角情况都做成完整可证明的通用算法。

## 13. 缓存线到底是怎么选的

缓存线选择集中在 `TrackLineManager.GetCacheLine()` 和 `CacheStrategy_simple`。

这块逻辑非常能代表当前实现的工程风格：不是全局求最优，而是多条规则叠加的启发式评分。

### 13.1 先看几个硬过滤

在真正评分前，会先做几层硬过滤：

- 必须是 `IsCache=true`
- 必须 `IsCanArrived=true`
- 剩余容量必须放得下这批车
- 不能是当前正在操作的那条线

此外还会加一些业务过滤：

- 如果还有未称重车辆，机库线不能作为缓存线；
- 如果机走阶段还没处理完，某些含“未来还要走机走体系”的线路不能被当缓存线；
- 如果这批车本身就是某条目标线当前正需要的连续车组，而且目标线已清空、可达、容量也够，那么直接返回目标线，不再进普通缓存评分。

### 13.2 机走体系的特殊捷径

如果某辆车目标属于机走/修线/卸轮等特殊体系，而且满足条件，缓存选择会直接倾向：

- `机走`

这说明机走线在当前实现里同时兼任两种角色：

1. 业务目标线
2. 特殊缓存中转线

### 13.3 `CacheStrategy_simple` 的评分思路

真正评分时，大致考虑这些因素：

- 距离成本：离当前线越近越好
- 空线奖励：空线更好
- 非空惩罚：已有杂车的线更差
- 同批次粘性：上一次就用过的缓存线有奖励
- 机走线惩罚：机走线不是轻易就用
- 功能线保护：喷漆、调梁等功能线若不是这批车的目标，会被施加大惩罚

你可以把它理解成一句人话：

> 能用空的、近的、不影响关键业务的普通缓存线，就别去占功能线和机走线。

### 13.4 注意：距离矩阵在当前实现里主要就用在这里

这点必须单独强调。

当前 C# 主实现虽然读了 `DistanceMatrix`，但从代码使用点看，距离基本就是在：

- 缓存线选择时算一个启发式成本

它并没有进一步输出：

- 每一钩经过了哪些股道片段
- 经过了哪些分支
- 实际路径长度是多少
- 道岔扳动了多少次

所以不要误以为当前实现已经实现了新版业务文档里那种“完整路径级”的调车计划。

## 14. 为什么 `机走` 会这么复杂

这部分单独讲，因为这是当前仓库里最重的业务分支。

### 14.1 `机走` 不是一条普通目标线

从代码行为看，`机走` 更像一个阶段性汇聚区。

很多去：

- 修1
- 修2
- 修3
- 修4
- 卸轮线
- 机走自身

这一类目标的车，会先被系统集中到机走体系里，再进入下一阶段处理。

### 14.2 当机走阶段启动后，系统会“收紧全场”

`PutCarToJZLine` 进入关键阶段后会做两件事：

1. 把相关目标线优先级拉到很高；
2. 暂停很多别的任务。

这其实是在表达一个很强的业务判断：

> 一旦进入机走汇聚阶段，就先把这拨事集中做完，别让别的普通线把节奏打散。

### 14.3 喷漆 / 洗罐为什么会受它影响

`TrackLineManager` 里写死了一个规则：

- 机走线上一旦有车，喷漆和洗罐就设为不可达；
- 机走线清空后，再重新开放。

这说明当前实现把它们之间的干扰关系当成了硬编码业务规则。

换句话说，不是通过完整拓扑推出来“会冲突”，而是：

> 工程上直接约定：机走占住时，喷漆和洗罐先别碰。

### 14.4 机走阶段完成后会发生什么

当 `PutCarToJZLine` 认为：

- 机走线上已经达到一定数量，
- 或者所有属于机走体系的剩余车都处理完了，

它会：

- 暂停所有任务
- 移除取车任务
- 开放修理线
- 把 `Constraint.IsPutJZFinished` 设为 `true`

这相当于告诉系统：

> 第一阶段的汇聚工作结束了，现在进入修线/后续放车阶段。

### 14.5 存4和机走的联动

`BackwardConstructionAlgorithm.OnTaskCompleted()` 里还有一个特殊联动：

- 如果 `Put_存4线` 完成，
- 就把机走线重新开放，
- 把机走线上所有车整体取回机车，
- 恢复任务，
- 移除取车任务。

这说明在当前实现里，`存4` 也不是普通线，而是被嵌进了机走相关的阶段切换逻辑里。

## 15. 事件机制为什么很重要

如果只看 `Run()` 主循环，很容易忽略事件机制的存在。

但实际上，当前实现里很多关键动作都不是主循环显式写出来的，而是事件驱动的。

### 15.1 某条线当前车清空时

可能触发：

- 创建 `Put_xxx`
- 广播给其他线“我已经空了”
- 重新打开部分本来不可达的线

### 15.2 某条线目标车取完时

可能触发：

- 如果线本身已空，直接创建 `Put_xxx`
- 如果线里还有杂车，先创建 `Clear_xxx`，再创建 `Put_xxx`

### 15.3 某条线完成时

可能触发：

- 把这条线重新标成缓存线
- 如果只剩机走未完成，开放相关线可达

所以整个系统的运行不是“算法单线程在推”，而是：

> 算法每做一步，线路状态就会反过来影响下一轮任务集合。

## 16. 输出结果到底是什么样

### 16.1 当前核心输出

当前算法最终返回的是 `List<Operation>`。

动作类型只有 3 种：

- `Get`
- `Put`
- `Weigh`

每条操作会记录：

- 这次搬了哪些车
- 当时机车上有哪些车
- 相关线前后状态

### 16.2 `OperationManager` 还会做动作合并

如果连续两条操作：

- 动作类型相同
- 线路名相同

它会自动合并。

这样输出更贴近现场的“成组操作”，而不是机械地一车一条记录。

### 16.3 Console 会再做一次面向人类的整理

`RailwayStation.Console/Program.cs` 里的 `_DealResult()` 会：

- 把 `TrainCars` 反过来
- 对 `Put` 动作把 `MoveCars` 也反过来

这是因为算法内部很多顺序是按“栈顶逻辑”存的，直接给人看会很反直觉。

所以离线导出的结果其实已经做过一层“人类视角翻译”。

## 17. 当前实现和新版福州东业务文档，到底是什么关系

这一节非常关键，因为很多人接手时会默认认为：

> 既然仓里已经有《福州东车辆段业务说明》和《福州东调车业务说明》，那当前 C# 代码应该已经完全按那两份文档实现了。

实际上不是。

### 17.1 一致的地方

二者在这些方向上是一致的：

- 都把问题看成“从当前分布到目标分布”的调车计划生成；
- 都强调目标顺序不能乱；
- 都重视长度约束、重车约束、称重插入；
- 都承认机车本身是计划状态的一部分；
- 都承认缓存线/临停是调车过程中不可避免的中间手段。

### 17.2 当前代码仍然明显更“旧”的地方

当前代码仍停留在比较粗粒度的股道模型上，例如：

- 还是 `修1 / 修2 / 修3 / 修4 / 机走 / 调梁 / 喷漆 / 洗罐` 这一类旧命名；
- 还没有真正切到 `targetMode / targetAreaCode / targetSpotCode` 那种合同；
- 很多台位、区域、作业区的细粒度语义还没有体现在主算法结构里。

### 17.3 当前代码没有真正实现的地方

从代码使用点看，下面这些新版文档里非常重要的能力，在当前 C# 实现里还没有真正落地成主输出：

- 每一钩完整经过路径
- 路径级合法性校验
- 道岔扳动次数作为显式优化目标
- `pathTracks` 之类的可回放路径结果
- 面向区域/台位的统一目标模式

所以更准确的说法应该是：

> 当前 C# 主实现是一个历史上形成的、可运行的调车规则引擎；新版福州东文档则更像下一代、更加严格的业务合同和建模方向。

## 18. 这套实现最值得记住的优点

虽然它不是一个“优雅的全局最优求解器”，但它有几条非常实用的优点。

### 18.1 工程上能跑

它不是纸面模型，而是已经把：

- 输入读取
- 容量约束
- 重车约束
- 称重插入
- 缓存线选择
- 阶段切换

这些都打通了。

### 18.2 对“连续车组”抓得很准

这点非常重要。

它没有傻傻地把所有车都当独立原子，而是明确识别：

- 连续车组
- 当前机车上是否已经形成某条目标线的连续段

这让它在很多场景下能比较自然地接近“按一钩成组处理”的现场思维。

### 18.3 事件 + 任务的写法很适合表达业务例外

像：

- 机走堵住后某些线关闭
- 某线清空后自动开放别的线
- 某线目标取完后自动生成放车任务

这种逻辑如果全堆在主循环里会非常乱。

现在拆成事件和任务后，虽然仍然复杂，但至少复杂度被分散了。

## 19. 这套实现最明显的局限和技术债

这部分也必须说透，不然后面继续演进时很容易误判。

### 19.1 它不是严格全局优化器

当前实现的核心决策 `_GetBestTarget()` 明显是启发式贪心。

它追求的是：

- 当前局面下看起来最顺手的下一步

而不是：

- 从全局证明总钩数最少、道岔最少、路径最短

### 19.2 特殊业务分支很多是硬编码

例如：

- 机走和喷漆/洗罐的联动
- 修线和存4的阶段切换
- 初始化就写死的一批优先线路

这让代码能快速表达业务，但也让它更难泛化。

此外，仓库里还存在明显的新旧命名混用。例如缓存策略里保护功能线时仍保留了 `喷漆库`、`调梁库` 这类名字，而主枚举和主逻辑里更多使用的是 `喷漆`、`调梁`。这种不一致不一定马上报错，但会让某些“本该生效的业务保护”悄悄失效或变弱。

### 19.3 还有一些“没遇到就先不实现”的分支

代码里能看到一些典型信号：

- `throw new Exception("目前没有遇到这样的数据，还未实现")`

这说明当前实现依赖已知样例集，对极端状态的完备性还不够强。

### 19.4 路径语义还没真正进入主算法

虽然有 `DistanceMatrix`，但距离主要只是缓存评分因子。

也就是说，当前系统最强的是：

- 编组重排与业务阶段调度

而不是：

- 细粒度走行路径规划

## 21. 最后用一段话总结这套实现

如果非要把当前 `RailwayStation` 的调车逻辑压缩成一句尽量准确的话，我会这么说：

> 它是一套以“目标编组顺序正确”为核心、以机车为动态中转栈、以任务系统管理业务例外、以连续车组和缓存启发式提升效率的调车规则引擎。

再说得更直白一点：

- 它最擅长的是“怎么把一堆车按顺序重排到目标线”；
- 它已经把很多现场常见的中间麻烦事编码进去了；
- 它现在还不是一套完整的路径级、台位级、全局最优钩计划求解器；
- 但它已经很好地展示了这个项目当前真正能跑的调车思路。

如果你后面要继续演进到新版福州东业务合同，那么最重要的不是推翻这套代码，而是先看清楚它现有的强项在哪里：

- 栈式编组重排
- 连续车组识别
- 任务化例外处理
- 机走阶段控制

这些都是以后无论换成更优雅的搜索算法、CP-SAT、块级 A*，还是更规范的福州东新合同实现时，仍然值得保留下来的经验资产。
