# fzed_shunting 对位车辆处理机制详解

本文说明当前仓库里的 `fzed_shunting` 求解器如何处理“对位车辆”。

这里的“对位车辆”主要指两类车：

- 精确台位车：必须停到明确台位，例如 `101`、`205`、`调棚:1`。
- 需要占用作业/库内台位的车：虽然不一定指定唯一台位，但进入大库、调棚、洗南、油、抛等位置后，也必须分配一个可用台位。

一句话概括：

> 当前系统不是只判断“车到没到某条股道”，而是同时维护一张 `车辆 -> 台位` 的占用表。对位车只有到达正确股道并拿到正确台位，才算真正完成。

---

## 1. 先理解“股道”和“台位”的区别

股道是大的位置，例如：

- `修1库内`
- `修2库内`
- `调棚`
- `洗南`
- `油`
- `抛`

台位是股道里面更细的位置，例如：

- `101` 表示 `修1库内` 的 1 号台位。
- `205` 表示 `修2库内` 的 5 号台位。
- `调棚:1` 表示调棚作业区的 1 号作业位。
- `洗南:2` 表示洗南的 2 号作业位。

所以，对位车的判断不能只看：

```text
这辆车最后是不是在修1库内？
```

还要看：

```text
这辆车在修1库内时，占到的是不是 101？
```

当前代码用 `ReplayState.spot_assignments` 保存这张表：

```python
spot_assignments = {
    "车号A": "101",
    "车号B": "调棚:1",
}
```

---

## 2. 输入里怎样表达对位目标

当前仓库同时支持新合同字段和旧字段兼容。

### 2.1 新合同：`targetMode=SPOT`

最明确的写法是：

```json
{
  "vehicleNo": "A001",
  "targetMode": "SPOT",
  "targetTrack": "修1库内",
  "targetSpotCode": "101"
}
```

归一化后，这辆车会变成：

```text
target_mode = SPOT
target_track = 修1库内
allowed_target_tracks = [修1库内]
target_spot_code = 101
```

也就是说，求解器知道它必须进入 `修1库内`，并且最终台位必须是 `101`。

如果是调棚、洗南、油、抛这类作业位置，也可以用 `SPOT` 表达，例如：

```json
{
  "targetMode": "SPOT",
  "targetTrack": "调棚",
  "targetSpotCode": "1"
}
```

归一化时会转成内部台位码：

```text
target_spot_code = 调棚:1
```

### 2.2 旧字段兼容：`isSpotting` 填数字

老输入里常见写法是：

```json
{
  "targetTrack": "大库",
  "isSpotting": "101"
}
```

这种也会被解析成精确台位目标：

```text
target_mode = SPOT
target_track = 修1库内
target_spot_code = 101
```

台位号第一位决定库线：

| 台位号 | 对应股道 |
|---|---|
| `1xx` | `修1库内` |
| `2xx` | `修2库内` |
| `3xx` | `修3库内` |
| `4xx` | `修4库内` |

例如：

- `101` -> `修1库内`
- `205` -> `修2库内`
- `307` -> `修3库内`
- `402` -> `修4库内`

### 2.3 `isSpotting=是` 不是精确台位

`isSpotting=是` 的意思不是“给我某个库内台位”，而是“这个作业区需要对位作业”。

例如：

```json
{
  "targetTrack": "调棚",
  "isSpotting": "是"
}
```

会变成：

```text
target_mode = AREA
target_area_code = 调棚:WORK
```

最终进入调棚时，系统会自动分配 `调棚:1` 到 `调棚:4` 中可用的作业位。

洗南、油、抛同理：

| 目标股道 | 可分配台位 |
|---|---|
| `调棚` | `调棚:1` 到 `调棚:4` |
| `洗南` | `洗南:1` 到 `洗南:3` |
| `油` | `油:1` 到 `油:2` |
| `抛` | `抛:1` 到 `抛:2` |

注意：上表是当前代码里的旧实现表，不等同于今天重新确认的“从北到南编号、按南端窗口定义对位区”的业务口径。二者的差异见本文第 17 节。

### 2.4 `isSpotting=迎检` 会打开迎检模式

如果有车辆写了：

```json
{
  "targetTrack": "大库",
  "isSpotting": "迎检"
}
```

整份计划会进入 `INSPECTION` 模式。

普通模式下，库内每条线只使用 5 个台位：

```text
修1库内: 101-105
修2库内: 201-205
修3库内: 301-305
修4库内: 401-405
```

迎检模式下，会扩展到 7 个台位：

```text
修1库内: 101-107
修2库内: 201-207
修3库内: 301-307
修4库内: 401-407
```

所以 `106`、`107`、`206`、`207` 这类台位，只有在迎检模式下才允许使用。

---

## 3. 输入校验会提前拦住明显错误

对位目标在归一化阶段会先做一层校验，避免错误输入进入求解器。

典型规则包括：

- `targetMode=SPOT` 必须提供 `targetSpotCode`。
- 库内台位号必须是 3 位，例如 `101`，不能是 `1` 或 `10`。
- 台位号第一位只能是 `1`、`2`、`3`、`4`。
- `101` 只能对应 `修1库内` 或泛目标 `大库`，不能写成 `targetTrack=修2库内`。
- 普通模式下不能使用 `106/107/206/207/...` 这类迎检扩展台位。
- 调棚、洗南、油、抛的作业位必须在允许范围内。

例如：

```json
{
  "targetMode": "SPOT",
  "targetTrack": "修2库内",
  "targetSpotCode": "101"
}
```

会被拒绝，因为 `101` 属于 `修1库内`，和 `修2库内` 不一致。

---

## 4. 初始状态也会分配台位

求解不是从一张空白台位表开始的。

`build_initial_state(...)` 会根据车辆当前所在股道和目标类型，初始化：

```text
track_sequences
spot_assignments
```

例如，当前已有两辆车在 `修1库内`：

```text
TD1 order=1
TD2 order=2
```

如果它们本来就应该在 `修1库内` 停放，那么初始状态会分配：

```text
TD1 -> 101
TD2 -> 102
```

这很重要，因为后面再有车想进 `101`，系统必须知道 `101` 已经被占了。

更准确地说，当前状态里的台位表不是“计划最终结果”，而是“现场此刻哪些台位被谁占着”。

---

## 5. 什么时候会分配台位

台位分配发生在车辆真正落到目标股道的时候。

当前系统的规则是：

> 台位是最终落位概念，临停过程不占台位。

举个例子：

一辆车最终要去 `101`，但中途先放到 `临1`。

这时不会给它分配 `101`，因为它还没到 `修1库内`。

等它真正 `DETACH` 到 `修1库内` 时，系统才调用 `allocate_spots_for_block(...)` 分配台位。

这样做的好处是：

- 临停线不会被错误地当成库内台位。
- 对位车可以先临停、再回库。
- 大库随机车和作业区车也可以正常经过中间股道。

---

## 6. 台位分配的基本规则

台位分配由 `allocate_spots_for_block(...)` 负责。

可以把它理解成一个“现场分座位”的函数：

```text
给我一组车、目标股道、当前已占台位
我返回这组车各自应该占哪个台位
如果没有可用台位，就返回失败
```

它的大致流程是：

1. 看这组车里哪些车在当前目标股道需要台位。
2. 为每辆需要台位的车列出候选台位。
3. 去掉已经被别人占用的台位。
4. 去掉被其他精确对位车预留的台位。
5. 选择第一个可用台位。
6. 如果任何一辆需要台位的车找不到台位，这个候选动作失败。

不同类型车辆的候选台位不同：

### 6.1 精确台位车

如果目标是 `SPOT`，候选台位只有一个。

例如：

```text
A001 -> 101
```

它到 `修1库内` 时，候选台位就是：

```text
[101]
```

如果 `101` 被占，动作失败，除非先把占位车移走。

### 6.2 大库随机车

如果目标是 `大库:RANDOM`，候选台位是目标库线上的空台位。

短车优先 `修1库内`、`修2库内`，必要时再用 `修3库内`、`修4库内`。

长车优先 `修3库内`、`修4库内`。

但如果某个台位已经被精确对位车预留，例如 `101`，随机车不会抢占这个台位。

### 6.3 作业区车辆

如果目标是调棚、洗南、油、抛这类作业区，会分配对应作业位：

```text
调棚:1
洗南:1
油:1
抛:1
```

如果第一个被占，就尝试下一个。

### 6.4 称重车

称重车在进入 `机库` 时，会占用一个特殊台位：

```text
机库:WEIGH
```

车辆进入机库后，会被加入 `weighed_vehicle_nos`，后续才按最终目标继续走。

---

## 7. 对位车什么时候算完成

最终判断在 `goal_is_satisfied(...)`。

对精确台位车来说，满足条件不是“到了目标股道”。

它必须同时满足：

```text
当前股道在允许目标股道内
并且 spot_assignments[车号] == target_spot_code
```

例如：

```text
车 A001 目标: 修1库内 101
```

下面这些状态都不算完成：

```text
A001 在 修1库内，但没有台位分配
A001 在 修1库内，但分到 102
A001 在 修2库内，哪怕分到了 201
A001 在 临1，中途临停
```

只有下面这种才算完成：

```text
A001 在 修1库内
spot_assignments["A001"] == "101"
```

这也是为什么验证器最后会报：

```text
final track/spot/weigh state does not satisfy goal
```

因为它检查的是“股道 + 台位 + 称重状态”的组合结果。

---

## 8. 挂车时会释放原台位

如果一辆车原来在库内并占着台位，后来被机车挂走，那么它的原台位会释放。

流程是：

1. `ATTACH` 从源股道北端取走一段车。
2. 这段车从 `track_sequences[source]` 中删除。
3. 这段车在 `spot_assignments` 里的台位记录也被删除。
4. 这些台位重新变成可用。

这符合现场逻辑：

```text
车离开了 101，101 就不应该继续被它占着。
```

---

## 9. 摘车时会尝试占用目标台位

`DETACH` 把机车后挂的尾部车组放到目标股道北端。

如果目标股道需要台位，系统会立刻尝试分配：

```text
DETACH 到 修1库内
  -> 尝试分配 101/102/...
```

如果分配成功：

```text
spot_assignments 更新
```

如果分配失败：

```text
这个 DETACH 动作不合法
```

所以求解器不会生成“看起来车到了库线，但其实没有台位”的动作。

---

## 10. 如果目标台位被别人占了，系统怎么办

这是对位处理里最关键的一块。

假设：

```text
A001 目标是 101
B002 当前占着 101
```

系统不会让 A001 直接进 `修1库内` 抢占 `101`。

它会先识别：

```text
101 的当前占用车是 B002
B002 挡住了 A001 的精确对位目标
```

然后生成“腾挪占位车”的候选动作：

1. 找到 B002 所在股道。
2. 看 B002 在这条股道的第几位。
3. 因为只能从北端取车，所以需要取走从北端到 B002 的整段前缀。
4. 把这段车临时移到可用临停线或其他候选位置。
5. 释放 `101`。
6. 后续再让 A001 进入 `修1库内` 并占用 `101`。

如果 B002 正好在股道北端，只需要移走 B002。

如果 B002 被前面的车挡着，就必须把前面的车一起拿出来。

这就是为什么对位车可能会带来额外调车钩数：不是系统不知道目标，而是目标台位在物理上可能被别的车挡住或占住。

---

## 11. 系统会避免再次堵住精确台位

求解器里有专门的精确台位评分逻辑。

它会鼓励：

```text
释放精确台位
```

也会惩罚：

```text
刚释放的台位又被别的车占住
把车重新放回会阻断精确台位的位置
```

这样做是为了避免一种低质量计划：

```text
第 1 钩：把占 101 的车移走
第 2 钩：又把另一辆随机大库车放进 101
第 3 钩：目标车还是进不了 101
```

代码里把这类动作称为：

- `exact_spot_clearance_bonus`：释放精确台位，加分。
- `exact_spot_reblock_penalty`：重新堵住精确台位，扣分。
- `exact_spot_same_track_repark_penalty`：同股道重放导致精确台位继续受阻，扣分。

这不是硬约束，因为有些复杂场景可能必须临时绕一下；但在搜索排序上，这些动作会明显靠后。

---

## 12. 大库随机车不会抢精确对位车的预约台位

假设一份计划里有：

```text
A001 目标 101
B002 目标 大库随机
```

即使 B002 比 A001 更早进 `修1库内`，系统也会知道：

```text
101 是 A001 的精确目标
```

所以 B002 的候选台位会排除 `101`。

如果 `101` 当前空着，B002 仍然不会占它。

这依赖 `exact_spot_reservations(...)`：

```text
收集所有精确台位目标，作为预留台位集合
```

然后在随机台位分配时跳过这些预留台位。

这条规则很重要，否则随机大库车很容易提前占掉精确对位车的位置，导致后面又要多出一轮腾挪。

---

## 13. 求解搜索如何把对位目标纳入整体计划

主搜索每一步会做三件事：

1. 根据当前状态生成合法 `ATTACH` / `DETACH` 候选动作。
2. 对每个候选动作执行一次状态更新。
3. 用启发式和评分决定哪些状态优先继续展开。

对位逻辑在这三步里都参与：

### 13.1 候选动作阶段

候选动作会先检查：

```text
这组车放到目标股道后，台位能不能分配成功？
```

如果不能，就不生成这个动作。

### 13.2 状态更新阶段

状态更新会同步改：

```text
track_sequences
loco_carry
spot_assignments
weighed_vehicle_nos
```

所以台位不是一个后处理结果，而是搜索状态的一部分。

### 13.3 搜索排序阶段

搜索会优先处理：

- 能让精确对位车完成的动作。
- 能释放精确台位占用的动作。
- 能减少未完成目标数量的动作。

并尽量推后：

- 重新堵住精确台位的动作。
- 无意义搬动已完成对位车的动作。
- 把已满足目标的车搬离目标位置的动作。

---

## 14. 回放和验证如何发现对位错误

求解完成后，计划还会经过回放和验证。

回放会从初始状态开始，一钩一钩执行：

```text
ATTACH
DETACH
ATTACH
DETACH
...
```

每一步都会更新 `spot_assignments`。

验证器最后检查每辆车：

```text
goal_is_satisfied(vehicle, final_track, final_state)
```

如果一辆精确台位车停错台位，即使股道正确，也会失败。

例如：

```text
目标: 101
最终: 修1库内 102
```

会被判定为不满足目标。

如果两辆车都想占 `101`，或者 `101` 已经被别人占用，新车再进来时台位分配会失败，回放阶段就会报错。

---

## 15. 一个完整例子

假设初始状态：

```text
修1库内:
  B002 -> 101

存5北:
  A001

A001 目标:
  修1库内 101
```

系统会这样理解：

```text
A001 是精确对位车
A001 必须进修1库内
A001 必须占 101
现在 101 被 B002 占着
```

因此直接做：

```text
存5北 A001 -> 修1库内
```

会失败，因为 `101` 不可用。

合理计划会类似：

```text
1. 从修1库内挂出 B002
2. 把 B002 临停到其他地方
3. 从存5北挂出 A001
4. 把 A001 摘入修1库内，占用 101
```

如果 B002 前面还有车挡着，那么第 1 步挂出的就不是 B002 一辆，而是“北端到 B002 的整段车”。

## 16. 当前实现的几个重要边界

### 16.1 台位分配是按当前代码内置表

库内台位表当前在 `domain/depot_spots.py` 中定义：

```text
修1库内: 101-105 / 迎检 101-107
修2库内: 201-205 / 迎检 201-207
修3库内: 301-305 / 迎检 301-307
修4库内: 401-405 / 迎检 401-407
```

作业区台位也在这里定义：

```text
调棚:1-4
洗南:1-3
油:1-2
抛:1-2
机库:WEIGH
```

这意味着如果现场台位数量或编号规则变了，需要同步更新这张表。现在已经确认作业线台位不是“局部作业区从 1 开始编号”，而是“整条线从北到南从 1 开始编号”，所以当前表只能代表旧模型，不能直接代表新业务规则。

### 16.2 对位是硬目标

对 `SPOT` 车辆来说，最终台位不对就是失败。

它不会因为“车已经到了目标股道”就放行。

### 16.3 临停不占台位

车辆临时放到临停线、存车线或中间股道时，不会提前占用最终台位。

只有进入真正需要台位的目标股道时才分配台位。

### 16.4 已完成车辆原则上不应轻易被搬走

评分逻辑会保护已经满足目标的车辆，特别是已经正确对位的车辆。

但如果它挡住了其他更高优先级目标，系统仍可能把它作为占位车临时移走，再后续恢复。

## 17. 按今天确认的新规则核对当前实现

今天确认后的核心变化是：

> 对位不是简单的“作业区前几个台位”，而是要按整条线路的北南方向编号，再从南端反推对位窗口。

这一节专门核对两件事：

- 序列方向是否正确。
- 台位候选集合是否和新规则一致。

### 17.1 挂车、摘车的序列方向：当前基本正确

当前代码把每条股道的车辆序列统一表示为：

```text
track_sequences[线路] = [北端车辆, ..., 南端车辆]
```

也就是说，数组越靠前，车辆越靠北；数组越靠后，车辆越靠南。

这个前提下，当前 `ATTACH` / `DETACH` 的顺序模型是这样的：

```text
ATTACH 挂车:
  只能从源股道北端前缀挂车
  也就是 source_seq[:n]

  北边的车先出来
  放到 loco_carry 序列靠前的位置
  可以理解为更靠机车、更靠里面

  南边的车后出来
  放到 loco_carry 序列靠后的位置
  可以理解为更靠外面
```

例如源股道从北到南是：

```text
[A, B, C]
```

挂出整段后，当前代码得到：

```text
loco_carry = [A, B, C]
```

其中：

- `A` 是北边先出来的车，在挂车序列靠里面。
- `C` 是南边后出来的车，在挂车序列靠外面。

摘车时，当前代码只允许摘 `loco_carry` 的尾部连续块：

```text
DETACH 摘车:
  只能摘 loco_carry[-n:]
  也就是挂车序列最外侧的一段
```

如果：

```text
loco_carry = [A, B, C]
```

那么可以先摘：

```text
[C]
[B, C]
[A, B, C]
```

不能直接摘：

```text
[A]
[A, B]
```

这和“摘车序列是尾部的先出”一致。

摘到目标线路时，当前代码把摘下来的车放到目标线路北端：

```text
target_seq = 摘下车辆 + 原目标线路车辆
```

也就是 prepend 到线路最北边。

需要注意一个细节：如果一次摘 `[B, C]`，真实瞬间可以理解为 `C` 这个外侧车先进入线路北侧，随后 `B` 这个更靠近机车的车再进入，并停在更北边。最终线路从北到南就是：

```text
[B, C, 原有车辆...]
```

当前代码直接把尾部块 `[B, C]` prepend 到目标线路，得到的最终物理顺序也是：

```text
[B, C, 原有车辆...]
```

所以它没有逐辆模拟“C 先出、B 后出”的中间瞬间，但对最终线路顺序是等价的。

结论：

> 挂车和摘车的序列方向当前是对的：源线路北端先挂，挂车尾部先摘，摘放结果加到目标线路北端，整条线路仍保持北到南顺序。

这部分对应的关键实现是：

- `build_initial_state(...)`：按 `order` 排出北到南序列。
- `_apply_attach(...)`：只允许取源股道北端前缀，并追加到 `loco_carry`。
- `remove_carried_tail_block(...)`：只允许摘挂车序列尾部。
- `_apply_detach(...)` / `replay_plan(...)`：把摘下车辆 prepend 到目标股道北端。
- `realign_spots_for_track_order(...)`：摘车后按整条目标线路当前北到南顺序重新对齐台位。

### 17.2 今天确认的新台位规则

新规则不需要先外部确认“每条作业线到底有多少个固定台位 `T`”。

原因是：对最终落位来说，作业线更像一条有长度的线，而不是一排预先切死的格子。车辆从北端放入，当前代码也是 prepend 到线路北端：

```text
新摘入车辆 + 原线路车辆
```

因此，先放进去且没有被移走的车辆，会在后续车辆继续从北端放入时相对变得更靠南；线路上原来已有的车辆，则会在新车放入后位于新车南侧。

举一个简单例子。

初始时调棚为空：

```text
[]
```

第 1 次放入 `A`：

```text
[A]
```

第 2 次再从北端放入 `B`：

```text
[B, A]
```

第 3 次再从北端放入 `C`：

```text
[C, B, A]
```

最终从南边数：

```text
A = 南边第 1 辆
B = 南边第 2 辆
C = 南边第 3 辆
```

所以，对位判断可以直接基于最终线路序列计算：

```text
线路序列按北 -> 南排列:
  seq = [北端车辆, ..., 南端车辆]

某车的北端序位:
  north_rank = seq.index(车号) + 1

某车的南端序位:
  south_rank = len(seq) - seq.index(车号)
```

容量是否允许，则不靠固定台位数，而靠长度判断：

```text
sum(线路上所有车辆 vehicleLength) <= 该线路 trackDistance
```

当前输入里本来就有每辆车的 `vehicleLength`，`trackInfo` / `tracks.json` 也有作业线路长度。例如当前主数据里：

| 作业线 | 当前有效长度 |
|---|---:|
| 调棚 | 174.3m |
| 洗南 | 88.7m |
| 油 | 124.0m |
| 抛 | 131.8m |

因此，真正需要建模的不是“固定 `T` 个台位”，而是：

```text
这辆车最终在线路上是从南边数第几辆
以及整条线路长度是否放得下
```

输入字段有 4 种口径：

| 输入值 | 业务含义 |
|---|---|
| `是` | 必须放到该作业线的对位窗口里 |
| `否` | 非对位，随便台位放置 |
| 空值 | 等同于 `否` |
| 具体台位号 | 必须放到整条线路上这个编号的台位 |

其中 `是` 要按“南端序位”判断：

| 作业线 | `isSpotting=是` 的对位要求 | 当前代码应计算的条件 |
|---|---|---|
| 抛丸 / 抛 | 最南边 2 个 | `south_rank in {1, 2}` |
| 油漆 / 油 | 最南边 2 个 | `south_rank in {1, 2}` |
| 洗罐 / 洗南 | 从南边数第 2 到第 4 个 | `south_rank in {2, 3, 4}` |
| 调棚 | 从南边数第 3 到第 6 个 | `south_rank in {3, 4, 5, 6}` |

`否/空` 的“随便台位”不是“不占台位”，而是：

```text
可以放到对位窗口
也可以放到对位窗口北边
也可以放到少量对位窗口南边
```

如果换成“南端序位”来理解，`否/空` 基本就是：

```text
只要最终在线路上
并且整条线路长度放得下
就可以
```

因为各线的非对位可放范围已经覆盖了整条可停车线路：

- 抛丸 / 油漆：对位窗口是南边第 1-2 辆，北边也可放，所以覆盖整线。
- 洗罐 / 洗南：对位窗口是南边第 2-4 辆，南边第 1 辆也可放，北边也可放，所以覆盖整线。
- 调棚：对位窗口是南边第 3-6 辆，南边第 1-2 辆也可放，北边也可放，所以覆盖整线。

所以在求解实现里，`是` 和 `否/空` 必须分成两套候选台位：

```text
isSpotting=是:
  目标股道正确
  且最终 south_rank 落在该作业线对位窗口内
  且线路长度放得下

isSpotting=否/空:
  目标股道正确
  且线路长度放得下
```

具体台位号要按整条线路的北到南编号解释：

```text
targetSpotCode = 1  -> 最终 north_rank == 1
targetSpotCode = 2  -> 最终 north_rank == 2
targetSpotCode = 3  -> 最终 north_rank == 3
```

如果最终线路上车辆数不足，或者线路长度放不下，则这个具体台位目标自然不能满足，不需要预先知道固定总台位数。

### 17.3 当前实现与新规则是否一致

结论先说：

> 大库普通模式大体一致；作业线不完全一致，尤其是 `否/空` 和具体台位号，目前和新规则差异较大。

逐项对比如下。

| 线路 | 今天确认的规则 | 当前实现 | 是否一致 |
|---|---|---|---|
| 抛 / 抛丸 | `是` = 最终南边第 1-2 辆；`否/空` = 线路内自由放置；具体数字 = 最终北边第 N 辆 | `是`、`否/空` 都基本落到 `抛:WORK`；候选只有 `抛:1`、`抛:2`；具体台位也只允许 `1/2` | 不一致：当前看固定候选码，不看最终南端序位 |
| 油 / 油漆 | `是` = 最终南边第 1-2 辆；`否/空` = 线路内自由放置；具体数字 = 最终北边第 N 辆 | `是`、`否/空` 都基本落到 `油:WORK`；候选只有 `油:1`、`油:2`；具体台位也只允许 `1/2` | 不一致：当前看固定候选码，不看最终南端序位 |
| 洗南 / 洗罐 | `是` = 最终南边第 2-4 辆；`否/空` = 线路内自由放置；具体数字 = 最终北边第 N 辆 | `是`、`否/空` 都落到 `洗南:WORK`；候选固定为 `洗南:1`、`洗南:2`、`洗南:3`；具体台位只允许 `1/2/3` | 不一致 |
| 调棚 | `是` = 最终南边第 3-6 辆；`否/空` = 线路内自由放置；具体数字 = 最终北边第 N 辆 | `是` 落到 `调棚:WORK`，候选 `调棚:1-4`；`否/空` 落到 `调棚:PRE_REPAIR` 单个伪台位；具体台位只允许 `1/2/3/4` | 明显不一致 |
| 大库 | 非对位可以放所有库内可用台位；对位按北到南 `n01` 到 `n05` | 普通模式每条修库线有 5 个台位，例如 `101-105`、`201-205`；随机大库车可在允许库线中分配；精确台位按具体数字匹配 | 普通模式基本一致；代码里的 `101-105` 相当于每条库线的 `n01-n05` |

大库还有一个需要单独确认的边界：

```text
当前代码保留了迎检模式：
  NORMAL:  n01-n05
  INSPECTION: n01-n07
```

也就是说，当前实现允许在 `isSpotting=迎检` 时使用 `106/107`、`206/207` 等扩展台位。今天的新口径只明确说“对位从北到南从 n01 到 n05”，没有提到迎检扩展是否继续有效。

因此大库判断应拆成两句话：

- 如果只看普通模式：当前大库规则基本一致。
- 如果新口径要求彻底取消 `n06/n07`：当前迎检模式就不一致。

### 17.4 当前不一致的根本原因

当前代码把作业线台位当成“作业区内部局部编号”：

```text
调棚:WORK = 调棚:1-4
洗南:WORK = 洗南:1-3
油:WORK = 油:1-2
抛:WORK = 抛:1-2
```

但今天确认的新规则要求把台位当成“最终线路序列里的物理序位”：

```text
整条线最终序列按北到南排列
对位窗口从南端反推
否/空使用更大的自由候选集合
具体台位号直接指向最终北端序位
```

这两个模型不是同一套模型。

更准确地说，当前实现并不是缺少固定台位总数 `T`，而是缺少下面这层判断：

```text
按目标线路最终北到南序列
计算每辆车的 north_rank / south_rank
再用 south_rank 判断“是”车是否处在对位窗口
再用 north_rank 判断具体台位号是否满足
```

线路能不能放得下，当前系统已有基础数据可以判断：

```text
车辆长度: vehicleLength
线路长度: trackInfo.trackDistance / tracks.effective_length_m
```

当前代码也已经在候选动作、目标 fallback 和最终验证里使用长度容量判断。缺口是：作业线对位判断还没有把“长度容量”和“最终南端序位”结合起来。

### 17.5 后续代码应该怎么改才通用

不建议继续用现在这种硬编码：

```text
调棚:WORK = 1-4
洗南:WORK = 1-3
油:WORK = 1-2
抛:WORK = 1-2
```

更稳的做法是把作业线台位从“固定候选码列表”改成“序位约束”。

可以把每条作业线定义成：

```text
线路: 调棚
容量: 使用 trackDistance / effective_length_m
编号方向: 最终序列北 -> 南
isSpotting=是: south_rank in {3,4,5,6}
isSpotting=否/空: 任意 south_rank，只要求线路长度放得下
targetSpotCode=N: north_rank == N
```

类似地：

```text
线路: 洗南
容量: 使用 trackDistance / effective_length_m
编号方向: 最终序列北 -> 南
isSpotting=是: south_rank in {2,3,4}
isSpotting=否/空: 任意 south_rank，只要求线路长度放得下
targetSpotCode=N: north_rank == N
```

然后由代码在每次模拟摘车后，基于目标线路新序列派生：

```text
new_seq = 摘入车辆 + 原线路车辆

对每辆车:
  north_rank = new_seq.index(vehicle_no) + 1
  south_rank = len(new_seq) - new_seq.index(vehicle_no)

再判断:
  是        -> south_rank 是否落入窗口
  否/空     -> 是否在线路上且长度放得下
  具体台位  -> north_rank 是否等于输入数字
```

这样做和现有序列模型是一致的，也能自然处理“先放的车后续被挤到更南边”的情况。

这类实现还需要注意一个业务后果：

```text
如果某车要求 洗南 isSpotting=是
它必须最终是南边第 2-4 辆
如果它被第一个放进空的洗南线，并且后面没有车在它南侧
它最终就是南边第 1 辆，不满足对位
```

因此求解器不能只在车辆刚摘入时判断一次，还要以最终线路序列为准。当前 `realign_spots_for_track_order(...)` 已经有“按整条目标线路顺序重算”的思路，后续应把作业线对位也改成同样的整线重算。

### 17.6 当前文档结论

当前仓库对“对位车辆”的机制是完整的：

```text
输入归一化
台位候选
台位分配
挂车释放
摘车占用
整线顺序重排
搜索评分
最终校验
```

但新确认的作业线台位业务口径还没有完全落到代码中。

准确地说：

- 序列顺序模型已经符合“北端先挂、尾部先摘、摘到北端”的物理口径。
- 大库普通模式基本符合 `n01-n05` 的对位口径。
- 作业线的 `是/否/空/具体台位` 还没有按“最终北到南序列 + 南端序位窗口 + 线路长度容量”建模。
- 当前 `spots.json` 和 `WORK_AREA_SPOTS` 只表达了旧的局部作业位数量，不足以支撑新规则。
- 不需要把“每条作业线固定有多少个台位 `T`”作为前置确认项；更通用的事实源是当前线路序列、车辆长度和线路长度。

## 19. 简单心智模型

最后可以用这个模型记住当前实现：

```text
输入阶段:
  判断哪些车是精确对位车，哪些车需要作业/库内台位

状态阶段:
  用 spot_assignments 记录当前台位占用

动作阶段:
  挂车释放原台位
  摘车尝试占用目标台位

搜索阶段:
  不能分配台位的动作直接不要
  目标台位被占时，主动生成腾挪占位车的动作
  释放精确台位的动作优先
  重新堵住精确台位的动作靠后

完成阶段:
  SPOT 车必须拿到指定台位才算完成
```

也就是说，当前仓库对“对位”的处理不是后处理，也不是输出阶段再修正，而是贯穿了输入、状态、动作生成、搜索排序、回放和最终验证的完整闭环。
