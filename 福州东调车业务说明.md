# 福州东调车业务说明

## 1. 文档定位

### 1.1 目的

本文档用于定义福州东车辆段调车计划的业务需求，包括输入合同、目标表达、合法性校验、线路与车辆约束、排位规则、输出合同和异常返回。

### 1.2 边界

1. 本文档只描述调车业务需求，不描述调车决策阶段、搜索阶段、旧代码适配层、缓存策略、内部能力组等实现逻辑。
2. 场区结构、线路分类、作业能力、工位数量等业务事实，以《福州东车辆段业务说明》为准。
3. 本文档只补充调车计划生成时必须额外明确的输入、约束、排位、输出和异常规则。

## 2. 基本作业口径

### 2.1 调车机位置与作业方向

1. 调车机固定在车列北侧进行取送作业。
2. 段内标准自动规划不允许把调车机放到车辆南端取送。
3. 无论现场习惯描述为“拉”还是“顶”，结构化输出中的车辆顺序一律按北到南表达。
4. 在调车的过程中对于起点或者终点在大库的车辆，放到勾计划的后半部分来调度。因为在调车开始的时候大库还可能还没有修完。
5. 若现场存在人工例外操作，不纳入当前 V1 自动规划范围。

### 2.2 一钩与勾计划

1. 一“钩”是调机完成一次完整“走行 + 连挂/摘解 + 放置”的最小作业单元。
2. 勾计划是若干连续“钩”组成的有序调车指令序列。
3. 每份计划都必须输出完整钩序、取送对象、起终点和经过路径。

### 2.3 优化目标优先级

按业务要求，调车计划的优化目标优先级固定为：

1. 总勾数最少。
2. 道岔扳动次数最少。
3. 路径总代价最小。
4. 每一钩都必须给出完整经过路径。

## 3. 输入合同

### 3.1 当前来源合同

```java
public class ShuntingAreaPlanInputVo {
    private List<AreaPlanTrackInfoVo> trackInfo; // 股道信息；每条记录对应 1 条股道及其本次计划使用的有效长度
    private List<AreaPlanExcelBo> vehicleInfo; // 车辆信息及调车需求；当前通常 1 条记录对应 1 辆车
    private String locoTrackName; // 调车机初始位置；当前可识别：空、机库、机北；空默认机库
}

public class AreaPlanTrackInfoVo {
    private String trackName; // 股道名；可传业务流程中的规范简称
    private BigDecimal trackDistance; // 股道有效长度，单位米
}

public class AreaPlanExcelBo {
    private String trackName; // 当前所在股道名；可传业务流程中的规范简称
    private String order; // 当前顺位；按北到南编号，归一化时必须可转整数
    private String vehicleModel; // 车型；当前至少包括 罐车、棚车、敞车、平车
    private String vehicleNo; // 车号；不能为空
    private String repairProcess; // 修程；当前可识别：段修、厂修、临修
    private BigDecimal vehicleLength; // 车辆真实长度，单位米
    private String targetTrack; // 本轮目标股道；可传业务流程中的规范简称，或聚合目标 大库 / 大库外
    private String isSpotting; // 兼容字段；当前可识别：空、否、是、迎检、101~107、201~207、301~307、401~407
    private String vehicleAttributes; // 兼容属性字段；当前只传空值或 1 个属性，已识别：重车、称重、关门车
}
```

### 3.2 输入字段闭环口径

1. 当前来源合同里没有独立的 `needAdjustBeam` 字段。
2. 调梁需求统一通过 `targetTrack=调棚` 与 `isSpotting` 联合表达：
   - `targetTrack=调棚` 且 `isSpotting=是`，表示进入 `调棚:WORK` 调梁作业区。
   - `targetTrack=调棚` 且 `isSpotting` 为空或 `否`，表示进入 `调棚:PRE_REPAIR` 调棚尽头预修区。
3. `vehicleAttributes` 当前一次只表达一个属性；如果后续业务需要同时表达多个属性，必须扩展为独立布尔字段，不再沿用单字符串兼容口径。

### 3.3 `isSpotting` 兼容解析规则

| `isSpotting` 原值 | 归一化结果 | 说明 |
| --- | --- | --- |
| 空、`否` | 默认 `targetMode=TRACK` | 默认只到线；若目标线属于必须表达作业区的线路，则按本文件第 4 章回填区域 |
| `是` | `targetMode=AREA` | 表示到目标线内的作业区，但不指定精准台位 |
| `101` 至 `407` | `targetMode=SPOT`，`targetSpotCode=原值` | 表示大库精准台位 |
| `迎检` | `yardMode=INSPECTION`，`targetMode=AREA` | 只用于大库随机区 |

补充规则：

1. 如果 `targetTrack` 属于 `调棚`、`洗南`、`油`、`抛`、`轮`、`大库`、`大库外` 或任一 `修N库内`，且 `isSpotting` 为空或 `否`，则不得继续按普通 `TRACK` 处理，必须按第 4 章默认回填为 `AREA`。
2. 如果 `isSpotting=是`，而 `targetTrack` 是普通存车线、临停线或走行线，则按非法输入返回。
3. 如果 `isSpotting` 是台位编码，则 `targetTrack` 必须能落到对应库内线；若上游给的是聚合目标 `大库` 或未给准，可按台位首位数字反推到 `修1库内` 至 `修4库内`。
4. 除上述四类值以外，其他值全部按非法输入返回。
5. `yardMode` 是整次计划级概念；若任一车辆使用 `迎检`，则整次计划按 `INSPECTION` 处理。
6. 同一批输入中如果出现互相冲突的计划级模式，按非法输入返回。

### 3.4 `vehicleAttributes` 兼容解析规则

| 原值 | 归一化结果 | 说明 |
| --- | --- | --- |
| `称重` | `needWeigh=true` | 需要经过机库称重工位 |
| `关门车` | `isCloseDoor=true` | 需要施加关门车专门排位约束；具体顺位规则见第 6 章 |
| `重车` | `isHeavy=true` | 参与牵引与分勾约束 |

补充规则：

1. 未识别属性不参与调车决策，按原值保留到原始字段。
2. `破车` 不再纳入当前 V1 属性范围。

## 4. 目标表达

### 4.1 三种目标模式

| 模式 | 含义 | 典型场景 |
| --- | --- | --- |
| `TRACK` | 只要求到达指定线，不要求精确作业区或台位 | 典型如一般存车线、集结线、机库、具体 `修N库外` |
| `AREA` | 要到达指定线内的指定作业区，但允许在线内自主落位 | 典型如调棚作业区、洗南作业区、大库随机区 |
| `SPOT` | 必须到达指定工位或台位 | 大库 `101` 至 `407` 精准台位 |

### 4.2 当前必须支持的区域编码

以下编码只定义调车计划中的结构化目标名称；对应的物理区段、能力边界和台位事实，以《福州东车辆段业务说明》为准。

| 区域编码 | 所属主线 | 含义 |
| --- | --- | --- |
| `调棚:WORK` | 调棚 | 调梁作业区，对应 `1` 至 `4` 号位 |
| `调棚:PRE_REPAIR` | 调棚 | 调棚尽头预修区 |
| `洗南:WORK` | 洗南 | 洗罐作业区 |
| `油:WORK` | 油 | 油漆作业区 |
| `抛:WORK` | 抛 | 抛丸作业区 |
| `轮:OPERATE` | 轮 | 卸轮/装卸运营区 |
| `大库:RANDOM` | 修1库内至修4库内 | 大库随机可放区域 |
| `大库外:RANDOM` | 修1库外至修4库外 | 大库外随机可放区域 |
| `机库:WEIGH` | 机库 | 机库称重工位，只能作为过程目标，不能由上游直接下发为最终目标 |

### 4.3 目标与输入的对应关系

1. 明确指定某一条可停线作为终点时，统一按 `TRACK` 处理，包括存1、存2、存3、存5北、存5南、机北、调北、洗北、预修、机棚、机库、存4北和具体 `修N库外`。
2. `存4南` 不是合法最终目标，只能作为过程临停线。
3. 调棚目标按 `isSpotting` 二分：`是 -> 调棚:WORK`，空或 `否 -> 调棚:PRE_REPAIR`。
4. 洗南、油、抛、轮目标在当前口径下都归一化为对应 `AREA` 目标，即 `洗南:WORK`、`油:WORK`、`抛:WORK`、`轮:OPERATE`。
5. 大库目标按粒度二分：未给精准台位时统一归一化为 `大库:RANDOM`；给了 `101` 至 `407` 时归一化为 `SPOT`，必要时再反推到对应 `修N库内`。
6. 上游若传聚合目标 `大库外`，统一归一化为 `AREA` + `大库外:RANDOM`；若直接传具体 `修1库外` 至 `修4库外`，则仍按第 1 条 `TRACK` 处理。
7. `机库:WEIGH` 只能由计划在过程内插入，不能由上游直接下发为最终目标。

## 5. 合法性校验与硬约束

### 5.1 输入合法性校验

生成计划前必须校验：

1. `trackName`、`targetTrack` 是否是已定义线路或合法聚合目标。
2. 同一股道内的 `order` 是否重复，且是否可转为整数。
3. 走行线是否被当作起点、终点或缓存点。
4. 临停线是否被当作正式终点。
5. `targetMode`、`targetTrack`、`targetAreaCode`、`targetSpotCode` 是否互相一致。
6. `targetMode=SPOT` 时，台位是否属于对应库内线。
7. 调车过程中所需的运行物理距离分支是否都已在《福州东车辆段业务说明》的运行物理距离附录中配置。

### 5.2 牵引与长度限制

V1 统一采用以下硬约束：

1. 纯空车最多 20 辆。
2. 混编重车最多 2 辆。#todo
3. 1 辆重车按 4 辆空车折算。
4. 经过 L1 道岔时，机后车辆总长度不得超过 190 米。

### 5.3 线路占用与工位约束

1. 线路占用统一按当前在线车辆 `vehicleLength` 之和计算。
2. 任一股道的已占用长度加本次拟送入车辆长度，不得超过该股道 `trackDistance`。
3. 调棚、洗南、油、抛、大库、轮线等同时受工位或台位数量约束，必须同时校验长度和工位数量。
4. 警冲标和岔区不能计入停车占用长度。

### 5.4 运行干涉与倒车距离

1. 运行干涉判断不使用 `trackDistance`，而使用《福州东车辆段业务说明》中的运行物理距离附录。
2. 若一钩包含倒车、折返或中间换线，必须校验倒车方向上剩余物理距离是否足够容纳“机车 + 本钩车辆集合”。
3. 运行和倒车允许经过警冲标和岔区；停车和留车不允许占用该部分。

### 5.5 称重约束

1. 只有 `needWeigh=true` 的车辆才允许安排称重步骤。
2. 单钩称重最多处理 1 辆称重车。
3. 称重只发生在机库称重工位，不改变该车最终业务目标。
4. 若车辆最终目标本身是 `机库`，则应先完成称重，再在机库线内完成最终停放。

## 6. 关门车规则

### 6.1 规则总表

| 场景 | 判定条件 | 约束 |
| --- | --- | --- |
| 不去 `存4北` | 本钩最终目标不是 `存4北`，且看本钩连挂完成后的机后总辆数 | 机后总辆数不超过 `10` 辆时不额外限制；超过 `10` 辆时，关门车不得位于第一顺位 |
| 去 `存4北` | 本钩最终目标是 `存4北` | 关门车在最终摆位结果中不得位于第 `1`、`2`、`3` 顺位 |

补充说明：

1. 本版规则不再要求为了关门车作业，先去 `存1`、`存2`、`存3` 单独挂取 `1` 辆车。
2. 所有关门车限制都按本钩最终落位结果判断，而不是按中间经过状态判断。
3. “机后总辆数”按本钩实际连挂完成后的机后车辆总数计算。
4. 若同一钩内同时包含多辆关门车，则每一辆关门车都必须分别满足对应顺位限制。
5. `存4北` 规则同时适用于把 `存4北` 作为修竣集结目标和作为追踪目的地的场景。

## 7. 特殊线路与排位规则

### 7.1 调棚

1. 调棚线路自北向南统一编号为 `1`、`2`、`3`、`4`。
2. `1`、`2`、`3`、`4` 均属于 `调棚:WORK` 调梁作业区。
3. `4` 号位以南尽头另有无编号预修区，对应 `调棚:PRE_REPAIR`。
4. 上游如何通过 `targetTrack=调棚` 与 `isSpotting` 表达目标，以及系统如何归一化，统一见第 3.2 节和第 4.3 节。

### 7.2 洗南、油、抛、轮

1. `洗南:WORK` 表示洗罐作业区，对应 `1`、`2`、`3` 号位。
2. `油:WORK`、`抛:WORK` 表示进入对应作业区，但当前不要求上游下发精准工位。
3. `轮:OPERATE` 表示轮线的运营作业区。

### 7.3 点到线 / 点到区域的默认落位规则

| 目标对象 | 默认落位规则 |
| --- | --- |
| 除 `机库`、`存4北` 外的普通 `TRACK` 目标 | 在线内随机分配 |
| `机库` | 在线内随机分配；若涉及称重步骤，执行顺序按第 5.5 节 |
| `存4北` | 在线内随机分配；关门车附加顺位限制见第 6 章 |
| `调棚:WORK` | 对位车辆进入 `1` 至 `4` 号工位 |
| `调棚:PRE_REPAIR` | 送入调棚尽头无编号预修区 |
| `洗南:WORK` | 对位车辆进入 `1` 至 `3` 号工位 |
| `油:WORK` | 送入油漆作业区内可用工位 |
| `抛:WORK` | 送入抛丸作业区内可用工位 |
| `轮:OPERATE` | 在线内随机分配 |
| `大库外:RANDOM` | 在 `修1库外` 至 `修4库外` 中选择可落位线路 |
| `修N库内` 且 `targetMode=SPOT` | 按指定台位落位 |
| `大库:RANDOM` | 按第 8 章自动分配大库台位 |

补充说明：

1. 普通 `TRACK` 目标的具体范围见第 4.3 节第 1 条。
2. 各作业区的物理边界、工位数量和台位事实，以《福州东车辆段业务说明》第 4 章和第 5 章为准。

## 8. 大库台位分配规则

本章只定义未指定精准台位时的自动分配规则，不重复定义大库物理台位本体；台位数量和编号范围以《福州东车辆段业务说明》第 5.4 节为准。

### 8.1 自动分配原则

1. 当前口径不再把段修、厂修写成两套固定物理区域。
2. 未指定大库精准台位时，仅按长度、模式和当前可用性分配。
3. `yardMode=NORMAL` 时，每库只使用 `01` 至 `05` 五个维修位。
4. `yardMode=INSPECTION` 时，每库可使用 `01` 至 `07` 七个维修位。

### 8.2 自动分配表

| 长度条件(米) | 允许台位 | 默认优先级 |
| --- | --- | --- |
| `>= 17.6` | `NORMAL` 时用 `301` 至 `305`、`401` 至 `405`；`INSPECTION` 时扩展到 `301` 至 `307`、`401` 至 `407` | 仅 3/4 库长车位 |
| `< 17.6` | `NORMAL` 时优先 `101` 至 `105`、`201` 至 `205`，不足时再到 `301` 至 `305`、`401` 至 `405`；`INSPECTION` 时扩展为优先 `101` 至 `107`、`201` 至 `207`，不足时再到 `301` 至 `307`、`401` 至 `407` | 先 1/2 库，再 3/4 库 |

### 8.3 自动分配的禁止项

1. `yardMode=NORMAL` 时不允许使用 `06`、`07` 台位。
2. 未指定台位时，不允许跳过长度规则直接放入不兼容台位。
3. 未满足入库条件时，不得直接把车送入库内台位。

## 9. 输出合同与异常返回

### 9.1 标准输出结构

标准输出不再复用输入兼容字段 `isSpotting`，统一输出归一化后的目标信息：

```java
public class HookPlanItemVo {
    private Integer hookNo;
    private String actionType;       // GET / PUT
    private String sourceTrack;
    private String targetTrack;
    private String isSpotting;      //  是否对位，规则同输入
    private Integer vehicleCount;
    private List<String> vehicleNos; // 按北到南
    private List<String> pathTracks; // 完整经过路径
}
```

字段约束：

1. `vehicleNos` 必须按北到南顺序返回。
2. `vehicleCount` 必须与 `vehicleNos.size()` 一致。
3. `pathTracks` 必须返回完整经过路径，不能只返回起终点。
4. `remark` 属于展示增强字段，不改变结构化主语义。
