# 调车计划生成 API 接口文档 (V3.1)

本接口用于根据车辆的初始状态、目标股道要求以及站场轨道容量，自动计算最优的调车作业序列及最终摆放位置。

---

## 1. 接口信息

* **接口地址**: `/api/plan/generate`
* **请求方法**: `POST`
* **数据格式**: `application/json`

---

## 2. 请求参数 (PlanCreatePostData)

请求体包含三个核心部分：起点状态、终点要求、轨道配置。

### 2.1 整体结构

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `StartStatus` | `List<StartCarDto>` | 是 | 车辆的初始位置和基本信息。 |
| `EndStatus` | `List<EndCarDto>` | 是 | 车辆的目标股道或区段要求。 |
| `TerminalLines` | `List<TerminalLineDto>` | 是 | 目标物理股道的最大台位容量配置。 |

### 2.2 StartCarDto (起点车辆信息)

| 字段名 | 类型 | 说明 | 样例 |
| --- | --- | --- | --- |
| `Line` | `string` | 起点位置全称（含区段名）。 | `修1库内` |
| `Position` | `int` | 起点连续台位号（从 1 开始）。 | `5` |
| `Type` | `string` | 车型。 | `C70` |
| `No` | `string` | 车号（唯一标识）。 | `1234567` |
| `Length` | `decimal` | 车辆换长（不填默认 11m）。 | `1.5` |
| `IsHeavy` | `bool` | 是否为重车（影响机车牵引限制）。 | `false` |
| `IsWeigh` | `bool` | 是否需要称重作业。 | `true` |

### 2.3 EndCarDto (终点要求信息)

| 字段名 | 类型 | 说明 | 样例 |
| --- | --- | --- | --- |
| `No` | `string` | 车号。 | `1234567` |
| `PossibleTargetLines` | `List<string>` | 候选目标区段列表（支持面到线退化）。 | `["调北", "调棚"]` |
| `Line` | `string` | 目标股道名。如果 `PossibleTargetLines` 为空，从此字段解析。 | `调梁` |
| `Position` | `int` | **非强制对位时**：用于动态计算所属区段；**强制对位时**：最终精确台位。 | `8` |
| `IsForceTargetPosition` | `bool` | 是否强制对位到 `Position` 指定的精确位置。 | `false` |

### 2.4 TerminalLineDto (轨道容量配置)

| 字段名 | 类型 | 说明 | 样例 |
| --- | --- | --- | --- |
| `Line` | `string` | 物理股道第一名称（First Name）。 | `调梁` |
| `MaxPosition` | `int` | 该物理股道允许规划到的最大连续台位号。 | `17` |
| `IsSpotting` | `bool` | 是否临检 | `true` |

---

## 3. 响应参数 (PlanGenerateResponseDto)

### 3.1 响应结构

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `Success` | `bool` | 请求是否处理成功。 |
| `Message` | `string` | 错误消息（失败时）。 |
| `StatusCode` | `int` | HTTP 状态码。 |
| `Data` | `PlanGenerateResultDto` | 规划结果数据。 |

### 3.2 PlanGenerateResultDto (结果详单)

* **`Operations`**: `List<Operation>`。按顺序排列的调车动作序列。
* `Action`: 动作类型（`Get`-取车, `Put`-存车, `Weigh`-称重）。
* `LineDisplayName`: 动作发生的股道展示名称。
* `MoveCars`: 本次动作移动的车辆列表。
* `TrainCars`: 动作完成后机车挂带的车辆状态。


* **`GeneratedEndStatus`**: `List<GeneratedEndCarDto>`。终点规划器生成的每辆车的精确最终台位。
* `DisplayLine`: 最终落位的展示名称（如 `调梁棚`）。
* `Position`: 最终精确台位号。
* `PositionMeters`: 折算的米数。



---

## 4. 业务逻辑说明

### 4.1 First Name 与 Second Name

为了兼顾算法求解和业务展示，系统内部使用两层名称：

1. **First Name (物理名称)**：如 `修1`、`调梁`。算法初始化、距离计算和轨道实体创建均使用此名称。
2. **Second Name (片段/区段名称)**：如 `库内`、`北`、`棚`。
* 用户输入时，可以在 `Line` 或 `PossibleTargetLines` 中使用 `修1库内` 或 `调北`。
* 系统会自动调用 `Terminal.ResolveTargetSegment` 将这些名称映射为物理股道及其对应的台位区间（如 `调北` 对应 `调梁` 的 1-6 位）。



### 4.2 对位逻辑

* **非强制对位**：用户提供的 `Position` 结合车辆换长，用于定位该车应落入哪个 `Second Name` 窗口（如落在库内还是库外）。
* **强制对位**：`Position` 将被视为不可更改的精确落位台位。
